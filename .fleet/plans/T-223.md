# Plan: Lua codegen + EVAL concurrency integration (T-223)

- **Issue:** #1070
- **Model:** opus
- **Date:** 2026-05-22
- **Epic:** #226 — see `~/.fleet/plans/issue-226.md` for full context
- **Blocked by:** #1068 (access traits), #1069 (Concurrency enum)

## Scope

New ticket surfaced by the Lua-driven ECS epic (#293) merging in
the interim. Without this, every Lua-defined system silently runs
serial after #1069 lands — a usability regression on the codegen
ergonomics just shipped.

## Affected files

### Codegen path

- `cmake/lua_codegen/parser.py` (or whatever the implementor of
  #586/#587 named it) — accept `concurrency = "..."` field in
  the `IRSystem.registerSystem({...})` Lua spec.
- `cmake/lua_codegen/emitter.py` — emit
  `Concurrency::PARALLEL_FOR` (or `SERIAL` / `MAIN_THREAD`) into
  the generated `createSystem<...>` call.
- Wherever the codegen tool tests live (likely `test/script/` or
  `test/codegen/`) — add cases for each enum value + a case for
  invalid string (`concurrency = "bogus"` should error at
  codegen time).

### EVAL path

- `engine/script/src/lua_script.cpp` — in `bindLuaDrivenEcs` (the
  runtime registration helper EVAL uses), force
  `Concurrency::MAIN_THREAD` regardless of the spec's value.
  Emit a one-time warning naming the system if the spec asked for
  `parallel_for`.
- `engine/script/CLAUDE.md` — extend the "Lua runtime + ECS
  modes" section. Document the `concurrency` field, the EVAL
  override, and the warning.

### Lua demo

- `creations/demos/lua_perf_grid/scripts/perf_grid.lua` (or the
  spec equivalent) — add `concurrency = "parallel_for"` to the
  wave-system spec.

## Approach

1. Read the codegen tool's spec-parser code (lands with #586 /
   #587). Find the existing field accepted set — `name`,
   `components`, `tick`, `mode`. Add `concurrency` to the
   accepted-field allowlist.
2. Map the string to enum: `"serial"` → `Concurrency::SERIAL`,
   `"parallel_for"` → `Concurrency::PARALLEL_FOR`,
   `"main_thread"` → `Concurrency::MAIN_THREAD`. Anything else
   errors at codegen time, naming the file + line.
3. Default missing field to `Concurrency::SERIAL` — every
   existing Lua spec keeps working unchanged.
4. EVAL: in `bindLuaDrivenEcs`, after the spec is parsed, if
   `concurrency` is anything but `MAIN_THREAD` or `SERIAL`,
   override to `MAIN_THREAD` and log one warning per system
   name. Sol2 is not thread-safe, LuaJIT GC isn't either — EVAL
   parallelism is structurally impossible until that's solved
   (separate epic).
5. The access-derivation trait from #1068 runs automatically on
   the codegen-emitted C++ lambda. A Lua spec with
   `concurrency = "parallel_for"` and an `EntityId` first param
   gets the same registration-time FATAL as a hand-written C++
   system. No Lua-specific validation needed.

## Acceptance criteria

- `lua_perf_grid` (CODEGEN mode) at 262K entities with
  `worker_threads = hw-2` matches `perf_grid` (C++) within ±10%.
  CODEGEN's whole point is native perf — threading must
  compose.
- `lua_perf_grid` (EVAL mode) with `concurrency = "parallel_for"`
  warns clearly and runs serial.
- Codegen tool errors on `concurrency = "bogus"`, naming the
  Lua file + line.
- A Lua spec with `concurrency = "parallel_for"` and
  `EntityId` first param fires the same registration FATAL as
  a hand-written C++ system. Unit-tested.

## Gotchas

- DO NOT expose `IRJobs::parallelFor` to Lua directly. Sol2
  calls from workers are unsafe; the only sanctioned Lua-side
  parallelism is the codegen path's per-system annotation.
- The CODEGEN tool already enforces a strict DSL subset (#587).
  `concurrency` is a metadata field on the registration spec,
  not a new code path inside the tick body. The DSL parser
  doesn't need any new tick-body rules.
- LuaJIT 2.1 (the EVAL runtime per #585) is single-threaded by
  design; even if sol2 were thread-safe, LuaJIT's GC and JIT
  state aren't. Don't try to fix this here.
- The EVAL warning fires once per system name, not once per
  registration call (some specs re-register on hot-reload).
  Use a `std::unordered_set<std::string>` to dedupe.

## Verification

- `fleet-build --target IRLuaPerfGrid` clean under
  `IR_LUA_ECS_DEFAULT_MODE=CODEGEN` (default).

  same lattice pattern as `IRPerfGrid` (C++).

  on all workers in CODEGEN mode.
- `fleet-build --target IRLuaPerfGrid` under
  `IR_LUA_ECS_DEFAULT_MODE=EVAL` builds; running it logs the
  one-time warning and the system runs serial.
- `IrredenEngineTest` 100% pass under both
  `IR_LUA_ECS_DEFAULT_MODE` values.
