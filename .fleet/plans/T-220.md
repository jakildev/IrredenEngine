## Scope

Pre-Phase-1 measurement surface for #226. No threading code in this PR —
this ticket lands the perf-matrix axis and the WorldConfig override that
the rest of the epic uses to validate `≥2× UPDATE throughput`. The cliff
is already documented by #1052; this fills in the before/after dimension.

## Affected files

- `scripts/perf/perf_grid_matrix.sh` — add `worker_threads` axis.
  Values: `0, 1, hw_concurrency()-2`. Plumbed through to the launched
  `IRPerfGrid` invocation (config flag, not env var, so it shows in the
  manifest).
- `scripts/perf/perf_summary.py` — add per-system UPDATE time columns.
  Stub per-worker utilization columns (filled by T-221's enkiTS hook).
- `scripts/perf/compare_perf_runs.py` — surface the new columns in the
  diff output so before/after compares against the threading axis.
- `engine/world/include/irreden/world/config.hpp` — add
  `entity_count_override` (`uint32_t`, default `0` = use creation's
  default). `worker_thread_count` does NOT land here; that's T-221.
- `creations/demos/perf_grid/main_lua.cpp` (or equivalent) — honor
  `entity_count_override` when set.
- `docs/perf-reports/threading_baseline.md` — new. `{4K, 32K, 262K} ×
  {0, 1, hw-2}` cells, all serial (no threading yet).

## Approach

1. Read `scripts/perf/perf_grid_matrix.sh` to see today's axis loop
   structure (zoom × subdivision sweep).
2. Add `WORKER_THREADS_LIST="0 1 $(($(nproc) - 2))"` (or detect via the
   existing build helpers). Wrap the existing nested loops with this
   third axis.
3. Pass `--worker-threads N` to `IRPerfGrid`. The exe parses it into
   `WorldConfig.worker_thread_count` — but since T-221 hasn't landed
   yet, the WorldConfig field doesn't exist yet either. Stub it:
   accept the flag, store it in the local config struct, ignore it
   downstream. T-221 wires the consumption side.
4. Build the matrix once, capture the baseline, write
   `threading_baseline.md`.

## Acceptance criteria

- `perf-gate.yml` on master passes with the new axis plumbed.
- `docs/perf-reports/threading_baseline.md` exists with all 9 cells
  filled. Numbers should be near-identical across the `worker_threads`
  axis today (since nothing's parallel yet) — this is the regression
  guardrail for T-221's link-enkiTS step.

## Gotchas

- Don't half-build the easy_profiler per-worker integration here.
  Stub the columns; T-221 owns that surface.
- `nproc` on macOS doesn't exist — use `sysctl -n hw.ncpu` in the
  matrix script, branched on `uname`.
- The `entity_count_override` field goes in `WorldConfig` next to
  `voxel_render_subdivisions`, not in a new `perf_grid` config struct.
  Keep the per-creation config surface lean.
