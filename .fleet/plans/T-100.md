## This PR: Lua-defined components with type inference

### Scope

Add a runtime path for Lua-defined components with a single native-SoA storage tier
and type inference. After registration, every field is a real native column — never a
per-frame `sol::table` lookup. If the engine cannot infer a native column type for a
field, registration fails at script-load time with a clear error.

### Deliverables

- `IComponentDataLuaTyped : IComponentData` in `engine/entity/`, holding a `std::variant`
  per declared field of: `int32_t`, `float`, `bool`, `std::string`, `sol::function`,
  bound usertypes, and (opt-in only) `sol::table`.
- `EntityManager::registerComponentDynamic(typeName, dataImpl)` parallel to the existing
  template `registerComponent<T>()`. Factor out shared bookkeeping (assigning a `ComponentId`,
  populating the pure-component-vectors map).
- `LuaScript::registerComponent(name, defaultsTable)` that:
  - Inspects each value in `defaultsTable` and infers the native field type per the Q1 table.
  - Accepts the explicit `{ field = {type="float", default=100} }` form for disambiguation.
  - **Raises a Lua error at registration** if any field has an unresolvable type, naming the field.
  - Allocates a real `ComponentId` via the dynamic registration path.
  - Identity rule (Q3): fails if `name` is already registered.
  - Auto-registers a field binding `("TypeName.fieldName", getter, setter)` into the
    modifier-framework registry so Lua-typed components participate in modifiers without extra code.
- Lua bindings: `IREntity.addLuaComponent`, `IREntity.getLuaComponent`, `IREntity.removeLuaComponent`.

### Verification

Smoke test: register `C_Hp = IRComponent.register("Hp", { current = 100, max = 100 })`,
attach to 100 entities, read back from a C++ test. Assert `Hp.current` is a real `int32`
column (cast through the column gives a native int, not a `sol::object`). Assert the
auto-registered field binding `"Hp.current"` is visible to the modifier resolver.
