## This PR: Lua-defined systems with archetype-batched dispatch

### Scope

Add a runtime path for Lua-defined systems. Lua systems iterate any mix of Lua-defined and
C++-defined components through the same `ComponentId` space. Dispatch is **archetype-batched**
(one `sol::function` invocation per matching archetype per tick), not per-entity.

### Deliverables

- `IRSystem::createSystemDynamic(name, includeIds, excludeIds, body)` taking a runtime
  component-id list and a `std::function<void(ArchetypeNode&)>` body. Lives next to the
  existing template `createSystem<...>()`; the templated path is unchanged.
- `system_manager.hpp` updated so the per-archetype dispatch path can invoke a runtime body
  alongside the templated tick path (likely a new `DynamicSystem` struct).
- `LuaScript::registerSystem({ name, components, excludes, tick })` that:
  - Resolves each component name to its `ComponentId` — Lua-registered types and C++ types
    in the creation's `lua_component_pack` resolve through the same id space (Q3).
  - **Fails fast** if an unbound C++ type is requested, with an error pointing at the component pack.
  - Builds the body wrapper: per matching archetype, expose each column as a typed Lua view
    (numeric arrays, usertype arrays for bound C++ types, etc.), and call the registered
    `sol::function` once per archetype.
- Returns a `SystemId` Lua can hold and drop into `registerPipeline` (T-102/PR 4).

### Verification

A Lua system iterates `(C_Position3D, C_Velocity3D)` — both C++-defined — alongside a
Lua-defined `C_Marker` tag, and writes to `C_Position3D`. The existing C++ render system
consuming `C_Position3D` picks up the change next frame. This proves shared-`ComponentId`
storage between C++ and Lua.

A separate test exercises a Lua system iterating only Lua-defined components (`C_Hp`, `C_Tag`).
Profiler shows one `sol::function` invocation per archetype per tick, not per-entity.
