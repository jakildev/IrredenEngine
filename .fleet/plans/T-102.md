## This PR: Pipeline composition + enum bindings + modifier-framework bindings

### Scope

Expose the engine's existing pipeline-composition surface and the already-landed modifier
framework to Lua. After this PR, a creation's entire `initSystems` can live in `main.lua`,
mixing engine systems, game systems, and Lua-defined systems in one heterogeneous pipeline
list. Lua scripts can also add modifiers to existing C++ component fields by name.

### Deliverables

- Bind `IRSystem::registerPipeline(event, {systemIds})` to Lua.
- Bind the existing `SystemName` enum and `GameSystemName` enum so Lua can spell
  `IRSystem.SystemName.LIFETIME` / `IRGameSystem.GameSystemName.GRID_BAKE`.
  Convenience: `IRSystem.systemId(SystemName.X)` → `SystemId`.
- New file `engine/script/include/irreden/script/lua_modifier_bindings.hpp` exposing:
  - `C_Modifiers` (vector-of-modifiers per entity)
  - the transforms enum (`ADD/MULTIPLY/SET/CLAMP_MIN/CLAMP_MAX/OVERRIDE`)
  - `C_ResolvedFields`
  - the field-binding registry (lookup-by-name and registration paths)
- Lua API: `IRModifier.add(entity, "Movement.speed", { transform = ADD, value = 0.5, source = src })`
  against any C++ engine/game component with a registered field binding. Lua-typed components
  are auto-registered (T-100/PR 2) so they participate without extra code.
- Sample creation whose entire `initSystems` lives in `main.lua` mixing engine, game, and Lua
  systems in one pipeline list.

### Verification

Lua calls `IRSystem.registerPipeline(IRTime.UPDATE, { IRSystem.systemId(SystemName.LIFETIME),
luaSystemId, IRGameSystem.systemId(GameSystemName.GRID_BAKE) })` and all three execute in
declared order. `IRModifier.add(entity, "Hp.current", { transform = MULTIPLY, value = 0.5 })`
against a Lua-defined Hp component results in `C_ResolvedFields` reflecting the halved value.
Running the sample creation produces the expected scene with no C++-side `initSystems`.
