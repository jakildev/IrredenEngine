# Plan: fleet — worktrees as work-item reservations, not role assignments

- **Issue:** #521
- **Model:** opus (foundation tasks), sonnet (role-doc plumbing), opus (migration)
- **Date:** 2026-05-07

## Scope

Decouple worktree identity from role identity. Today
`opus-worker-1` always runs opus-worker iterations and is implicitly
"reserved" for that role. We retire that coupling and replace it with
explicit per-work-item reservations:

- A worker iteration claims a task AND reserves its current worktree
  to that task.
- Reservation persists across iteration boundaries (rate-limit
  death, dispatcher restart, machine sleep).
- The next iteration on this pane resumes the reserved task in the
  same worktree, with all dirty state intact.
- On task completion (PR opened) or explicit abandonment, the
  reservation releases and the worktree returns to the free pool.

This eliminates the entire class of dirty-worktree leaks observed
on `opus-worker-1` after T-096 (PR #507) and on three other
worktrees in the same wave.

## Resolved design questions

### Q1: Concurrency cap by role

**Decision:** soft cap configured per role at fleet-up time, enforced
by the dispatcher. New `fleet-up` config field per role:

```
opus-worker:    concurrency=2
sonnet-fleet:   concurrency=4
sonnet-author:  concurrency=2
```

Dispatcher logic:

1. Count currently in-flight iterations per role (panes whose
   reserved task's role tag matches, plus any pane mid-claude-print
   for that role).
2. If `count >= cap`, defer dispatch.

Reservations themselves don't enforce role caps — they're orthogonal.
The cap protects credit budget; reservations protect dirty-state
continuity.

### Q2: Affinity vs flexibility

**Decision:** strict affinity. Once a task is reserved on worktree W,
only W can resume it. Migration of dirty state across worktrees is
fraught (uncommitted edits, untracked files, stash conflicts) and
the safety win from strict affinity outweighs the throughput loss.

Edge case: if W is "busy" (claude process running) when the
dispatcher fires the resumption signal, the dispatcher waits — does
not migrate. In practice this won't happen: a dispatcher fires when
the pane returns to bash, so the busy-check is implicit.

### Q3: Cross-repo worktrees

**Decision:** reservations key on `(repo, task_id)` pair, mirroring
`fleet-claim --repo game`'s slug namespacing. The reservation file
records both:

```json
{
  "task_id": "T-002",
  "repo": "game",
  "worktree": "worktree-1",
  "branch": "claude/T-002-...",
  "engine_or_game_path": "/Users/.../creations/game/.claude/worktrees/worktree-1",
  "claimed_at": "2026-05-07T22:13Z"
}
```

The reservation lives at `~/.fleet/reservations/<worktree>.json`
(per-worktree, not per-task) — one worktree owns at most one
reservation at a time, regardless of repo.

### Q4: Abandonment policy

**Decision:** never auto-wipe dirty state. Instead:

- After 24h, the staleness checker (extends existing
  `fleet-claim check-stale`) flags the reservation.
- The flag triggers an issue (`fleet:stuck-worktree` label) with the
  worktree name, task ID, branch, and a `git status --porcelain`
  capture.
- The human triages: either resumes manually, escalates to a fresh
  task, or wipes the worktree explicitly.

Silently losing in-progress work is strictly worse than the current
"stuck forever" state. Escalation is loud and explicit.

### Q5: Architects exempted

**Decision:** confirmed. `opus-architect` and `game-architect` keep
dedicated worktrees and the babysit lifecycle. They're interactive
panes, not transient workers — they don't fit the "iteration ends
on PR open" lifecycle that motivates reservations. Architects'
worktree names stay as-is (`opus-architect-1`, `game-architect-1`).

## Approach

Sequenced as a 6-task stack. PR-1 lands the data plane;
PR-2 makes the dispatcher reservation-aware; PR-3 plumbs role docs;
PR-4 renames worktrees; PR-5 adds escalation; PR-6 adds
concurrency cap. PRs 4 and 6 can land in parallel after PR-3.

### T-NNN-1 [opus]: fleet-claim worktree reservation primitives

**Files:**
- `~/bin/fleet-claim` (extend existing claim/release; add new
  reserve/release-worktree subcommands)

**New subcommands:**
- `fleet-claim reserve <worktree> <task-id> [--repo game]` — atomic;
  fails if worktree already reserved or task already claimed.
- `fleet-claim release-worktree <worktree>` — clears reservation.
  Idempotent.
- `fleet-claim worktree-for-task <task-id> [--repo game]` — prints
  worktree name holding the reservation, or empty.
- `fleet-claim reservation-of <worktree>` — prints task_id of the
  reservation, or empty.
- `fleet-claim list-reservations` — print all current reservations.

**Storage:** `~/.fleet/reservations/<worktree>.json` with
`task_id`, `repo`, `branch`, `claimed_at`, `worktree_path`. JSON
chosen for forward-compat (additional fields for future audit).

**Atomicity:** create-via-`mkdir` pattern matches existing
`CLAIMS_DIR/<slug>/` layout — but we use a single JSON file plus a
lock-directory sibling (`<worktree>.lock/`) for atomic
create-or-fail.

**Existing `claim` integration:** when an agent runs
`fleet-claim claim <task-id> <agent>`, the claim subcommand also
writes the reservation if `<agent>` matches a worktree name (the
common case). Backward compatible — agents that don't know about
reservations still work.

**Existing `release` integration:** `fleet-claim release <task-id>`
also clears the matching `worktree.json` if present.

**Tests:** integration test hitting all new paths; verifies atomic
create-or-fail; verifies stale-claim-clears-reservation.

### T-NNN-2 [opus]: dispatcher reservation-aware pane selection

**Files:**
- `~/bin/fleet-dispatcher`

**Logic change:** when firing a role-X iteration, dispatcher's
pane-selection order becomes:

1. **Reserved-pane check** — list reservations whose
   `task_id`'s `Model:` field matches role X's role tag (need a
   one-line reverse-lookup helper: `fleet-claim reservation-role
   <worktree>` → reads the .json, looks up the task in TASKS.md).
   If a reserved pane exists with this role, route there.
2. **Free-pane fallback** — current `@fleet-role` selection over
   panes with NO active reservation.
3. **Cap check** — if count of in-flight + reserved iterations
   already hits the configured cap, defer.

This step keeps `@fleet-role` as a fallback signal but lets
reservations take priority. After PR-4 renames worktrees, the
`@fleet-role` tag becomes purely advisory and could eventually be
dropped — out of scope for this issue.

**Tests:** spin a fake reservation via `fleet-claim reserve`;
verify dispatcher routes to the reserved pane; remove reservation;
verify it routes to the free pane.

### T-NNN-3 [sonnet]: role docs — startup reservation check

**Files:**
- `.claude/commands/role-opus-worker.md`
- `.claude/commands/role-sonnet-author.md`
- (sonnet-reviewer doesn't claim tasks — exempt)
- (queue-manager / merger don't claim tasks — exempt)

**Changes per role:**

- New step 0.5 between heartbeat and feedback-label scan:
  ```
  RESERVATION_TASK=$(fleet-claim reservation-of <your-worktree-basename>)
  if non-empty:
    branch=$(jq -r .branch ~/.fleet/reservations/<basename>.json)
    git checkout "$branch"   # already there if we crashed mid-iteration
    skip steps 2 (planning) and 3 (pickup) — jump to step 4 with this task
  ```

- Step 6 (work it) and step 11 (commit-and-push) are unchanged.

- Step 12 (reset): replace `start-next-task` reset with explicit
  `fleet-claim release-worktree <basename>` BEFORE
  `start-next-task`. Otherwise the resumption logic would route us
  back to the already-finished task.

- Step 8 (escalate): when posting `fleet:design-blocked`, the
  reservation stays — the next iteration RESUMES the same worktree.
  This is exactly the "dirty state continuity" win we want.

- Hard-rule update: drop the existing "Edit/Write paths must stay
  inside your worktree" caveat about parent-clone confusion — under
  the new model, an agent has no reason to think about other
  worktrees during execution; the reservation makes the binding
  explicit.

**Tests:** dry-run-mode role iteration with a stale reservation in
place; verify the role exits cleanly (no double-claim attempt) and
the reservation is preserved.

### T-NNN-4 [opus]: worktree naming migration

**Files:**
- `~/bin/fleet-up` (worktree provisioning)
- All role docs (drop role-name assumptions in startup-banner text;
  use `WORKTREE_BASENAME` env var or `pwd` basename uniformly)
- `docs/agents/FLEET.md` (workflow doc updates)
- `docs/agents/FLEET-CACHE.md` (state.json schema if needed)

**Naming:** existing names → generic:
- `opus-worker-1` → `worktree-1`
- `opus-worker-2` → `worktree-2`
- `sonnet-fleet-1..N` → `worktree-3..(2+N)`

Architects keep their names (`opus-architect-1`, `game-architect-1`).

**Migration script** (one-shot; runs once on the human's machine):
- Stops fleet (`fleet-down`).
- Walks `~/src/IrredenEngine/.claude/worktrees/` and the parallel
  game tree.
- For any worktree that's CLEAN: rename via `git worktree move`.
- For any worktree that's DIRTY: file an escalation issue listing
  the dirty files; the human resolves before re-running migration.
- Updates `~/.fleet/state/state.json` projection and `tmux` window
  names.

**Coordination:** PR-4 lands the new fleet-up provisioning code +
migration script; the human runs the migration locally (it touches
their machine state, not just code). The PR includes a
`docs/agents/MIGRATION-WORKTREES.md` walkthrough.

### T-NNN-5 [sonnet]: worktree reservation staleness escalation

**Files:**
- `~/bin/fleet-claim` (extend `check-stale` subcommand)
- `~/bin/fleet-up` (the wrapper that periodically calls check-stale)

**Logic:**
- For each reservation older than 24h:
  - Pull the dirty file count (`git -C <worktree_path> status
    --porcelain | wc -l`).
  - File a GitHub issue (engine repo) titled `fleet: stuck
    worktree <name> reserved for T-NNN`.
  - Body: reservation JSON, dirty file list, last commit on branch,
    timestamp.
  - Label `fleet:stuck-worktree` (new label — add to engine repo
    via human or bootstrap step).
  - Comment is one-shot per reservation (track via
    `~/.fleet/reservations/<name>.escalated` flag).

**Why never auto-wipe:** stated in Q4 above.

### T-NNN-6 [opus]: per-role concurrency cap

**Files:**
- `~/bin/fleet-up` (config schema; new lines per role)
- `~/bin/fleet-dispatcher` (cap-check at dispatch time)

**Config:** `~/.fleet/fleet-up.conf` (or wherever fleet-up reads
config) gains a per-role concurrency block. Default for opus-worker
is 2 (preserves current behavior). Human edits to tune.

**Dispatcher cap-check:** before dispatching role X to a free pane,
count `(in-flight + reserved)` for X. If `>= cap[X]`, defer.

**Why opus-only-needed-PR-6:** PR-2 already added a deferral
mechanism for dispatch when no free pane is available; this PR
extends it with the soft cap.

## Acceptance criteria

After all 6 tasks merge:

1. **Crash test:** kill an opus-worker mid-iteration with dirty
   state. The next dispatcher tick fires a fresh claude on the same
   pane; the new iteration detects the reservation, checks out the
   reserved branch, sees the dirty state, and continues.
2. **Clean lifecycle:** worker iteration claims T-NNN, makes edits,
   opens PR, runs `start-next-task`. The reservation clears. The
   worktree appears in `fleet-claim list-reservations` as empty.
3. **Cross-repo:** game-task workflow works identically — engine
   `fleet-claim claim` and game `fleet-claim --repo game claim` both
   wire through to reservation tracking with proper namespacing.
4. **Cap honored:** with `concurrency=2` for opus-worker, dispatcher
   never fires a third opus-worker iteration concurrently, even if
   3 worktrees are free.
5. **Stuck-worktree escalation:** a reservation older than 24h
   triggers exactly one issue under `fleet:stuck-worktree` label.
6. **Architect untouched:** opus-architect / game-architect panes
   continue using `fleet-babysit` lifecycle, are not affected by
   reservation logic.

## Gotchas

- **Existing claim files have no `worktree` field.** PR-1 includes a
  one-shot migration: scan `~/.fleet/claims/*/` and synthesize
  reservations from the `owner` file (which today contains the
  worktree basename). Backwards compat for in-flight tasks at the
  moment PR-1 lands.

- **`fleet-up clear-all` semantics.** Today wipes all claims. Under
  the new model, it should also clear `~/.fleet/reservations/` —
  but the dirty state in worktrees stays, which is correct (clear
  the metadata, leave the work). Document this in the migration
  doc.

- **Pane currently mid-claude.** `fleet-dispatcher` already handles
  "pane is busy"; the reservation-aware path inherits the same
  signal. No new race surface.

- **Architect worktrees in `fleet-claim list-reservations`.** They
  shouldn't appear because architects don't call `fleet-claim
  claim`. But the migration in T-NNN-4 might inadvertently create
  reservation entries for them. Filter them out by name in
  `list-reservations` output (architect names are well-known).

- **State.json schema.** Scout's projector might want to surface
  reservations in `state.json` so cache reads can show "what's
  reserved where" without separate lookups. Out of scope for this
  issue but flag it for a follow-up.

- **`~/bin/` is outside the repo.** Changes to `fleet-claim` /
  `fleet-dispatcher` need the human to either symlink from a tracked
  location or accept that those scripts ship via PR but install via
  hand. Worth a 1-comment thread with the human at PR-1 time.

- **Test coverage.** None of the fleet binaries have an automated
  test harness today. PR-1 should add a minimal bash-test harness
  (one of the human's stated preferences in past PRs).
