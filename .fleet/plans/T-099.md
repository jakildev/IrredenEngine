# Plan: Lua-driven ECS authoring (engine #293)

## Context

Today the engine's Lua surface is one-way: C++ defines components and
systems, then the `kHasLuaBinding<T>` + `bindLuaType<T>` traits selectively
expose them. Lua scripts can construct, mutate, and call methods on bound
types and register event-handler callbacks, but cannot define new component
shapes or new systems at runtime. Anything that needs a new component or
system tick has to be a C++ change with a rebuild.

The goal of #293 is to lift that ceiling: a creation should be able to
author **all** game-specific logic in Lua — new component types, new
systems, and the pipeline that schedules them — while still mixing freely
with C++ engine and game-defined systems.

---

## Design decisions (Q1–Q5)

### Q1: Component layout from Lua → single native-SoA storage tier with type inference

The author API still accepts the short form `IRComponent.register("Hp",
{ current = 100, max = 100 })`, but **the engine never stores those as
opaque Lua tables**. At registration time, the engine inspects each
default and infers a native field type:

| Default value                            | Inferred field type    |
|------------------------------------------|------------------------|
| Lua integer (`5`)                        | `int32`                |
| Lua float (`5.0`)                        | `float`                |
| Lua string (`"hello"`)                   | `std::string`          |
| Lua boolean (`true` / `false`)           | `bool`                 |
| Bound C++ usertype (`vec3.new(...)`)     | that usertype          |
| Bound enum value                         | the enum               |
| Lua function (`function() ... end`)      | `sol::function`        |
| Anything else (nested table, nil, etc.)  | **registration fails** |

**No silent fallback.** Unresolvable fields raise a Lua error at script-load
time naming the offending field. Authors use `{ field = {type="float", default=100} }`
to disambiguate; `{type="table", default={...}}` is the explicit opt-in escape hatch.

### Q2: System tick from Lua → archetype-batched dispatch, mandatory

One `sol::function` invocation per matching archetype per tick. Per-entity
dispatch is documented as a footgun (10-100× slower), not the default API.

### Q3: Archetype impact → one ComponentId per type, C++ and Lua share identity

Lua-defined components get real `ComponentId`s via the same lazy registration
path, routed to `IComponentDataLuaTyped` instead of `IComponentDataImpl<T>`.
No parallel archetype graph. C++ systems and Lua systems reading `C_Position3D`
access the same column byte-for-byte.

### Q4: Hot-reload → system bodies hot-reloadable; component schemas deferred to follow-up

`IRSystem::replaceSystemBody(systemId, newFn)` rebinds the `sol::function`.
No archetype changes, no entity migration.

### Q5: Modifier framework integration

Schemafied Lua components auto-register field bindings into the modifier
framework's string-keyed registry. Lua scripts can call
`IRModifier.add(entity, "Hp.current", { transform = ADD, value = 0.5 })`
against any C++ or Lua-defined component with a registered binding.

## Staged delivery (all 6 PRs)

### PR 1 — T-099: Design doc (`docs/design/lua-driven-ecs.md`)

Lock in Q1–Q5, public API shape, parity gate spec, out-of-scope follow-ups.
No code changes outside `docs/`.

### PR 2 — T-100: Lua-defined components with type inference

- `IComponentDataLuaTyped : IComponentData` in `engine/entity/`
- `EntityManager::registerComponentDynamic(typeName, dataImpl)`
- `LuaScript::registerComponent(name, defaultsTable)` — type-infer, allocate ComponentId, auto-register field bindings
- Lua bindings: `IREntity.addLuaComponent`, `getLuaComponent`, `removeLuaComponent`

### PR 3 — T-101: Lua-defined systems with archetype-batched dispatch

- `IRSystem::createSystemDynamic(name, includeIds, excludeIds, body)`
- `LuaScript::registerSystem({ name, components, excludes, tick })` — resolves component names to ComponentIds (same id space as C++)
- Returns `SystemId` for use in `registerPipeline`

### PR 4 — T-102: Pipeline composition + enum bindings + modifier-framework bindings

- Bind `IRSystem::registerPipeline(event, {systemIds})` to Lua
- Bind `SystemName` and `GameSystemName` enums; convenience `IRSystem.systemId(SystemName.X)`
- New `engine/script/include/irreden/script/lua_modifier_bindings.hpp` — bind `C_Modifiers`, transforms enum, `C_ResolvedFields`, field-binding registry
- Sample demo creation with pure-Lua `initSystems`

### PR 5 — T-103: Hot-reload of Lua system bodies

- `IRSystem.replaceSystemBody(systemId, newFn)` rebinds the `sol::function`
- Doc in `engine/script/CLAUDE.md`

### PR 6 — T-104: Lua port of perf_grid + parity gate

- New demo `creations/demos/lua_perf_grid/` — 64×64×64 voxels, wave animation, same render pipeline
- Parity gate: Lua wave-animation system per-tick cost ≤ 1.5× C++ `SystemPeriodicIdlePositionOffset`
- If gate fails: amend design doc with corrective decision (LuaJIT, codegen bodies, etc.) before further work

## Out of scope for #293

- Hot-reloading component schemas (mid-run archetype migration)
- Generic `IRLuaCreation` boot binary (purely additive follow-up)
- Lua-defined relation components
- LuaJIT migration (only if PR 6's parity gate fails)

## Performance budget

With native SoA storage (Q1) and archetype-batched dispatch (Q2):
- Archetype-batched Lua system, trivial body, native fields: target ≤ 2× C++ tick body
- Per-entity Lua dispatch (escape hatch only): still 30-100× C++ — documented footgun
- Explicit `sol::table` opt-in fields: one Lua table lookup per access per entity — author accepted cost explicitly

**Parity gate (PR 6):** Lua `lua_perf_grid` wave-animation per-tick cost ≤ 1.5× C++ `perf_grid` equivalent, measured via `IRProfile` with `profiling_enabled = true`.
