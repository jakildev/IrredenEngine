# Plan: scheduler — allow claiming blocked tasks via stacked PRs

- **Issue:** #501
- **Model:** opus
- **Date:** 2026-05-07

## Scope

Today the scheduler (sonnet-author and opus-worker pickup logic) only
considers tasks whose `Blocked by:` resolves to merged work or
`(none)`. Blocked-but-stackable tasks sit idle even when both opus
panes (and the sonnet pane) are otherwise free. The PR-opening
machinery already supports `--base <feature-branch>` via
`commit-and-push`'s **fleet-stack** and **cursor-stack** modes; the
gap is on the *claim* side and on the *cascade* side (merger /
reviewer behavior when the upstream PR churns).

This plan extends cross-author stacking into the autonomous
scheduler so a free worker can pick up `T-Y` while `T-Y`'s blocker
`T-X` still has an open PR. Goal: increase pane utilization without
sacrificing review correctness or merge ordering.

## Locked-in design decisions

These resolve the three open questions in the issue. Implementers
should follow them; further design tweaks need an architect comment
on the issue, not a unilateral change in the PR:

### Q1 — How aggressive should "claim blocked tasks" be?

**Decision: only-if-no-unblocked.** Two-tier pickup. A worker first
runs the existing filter (owner=free, `blocked_by` resolves to
merged/`(none)`). Only if that filter returns nothing does it fall
through to the stackable-blocked tier. Cross-author stacking has a
non-trivial coordination tax (rebase cascades, awaiting-base waits,
reviewer gating); we want the simple path preferred by default and
this path used only to recover otherwise-idle pane time.

### Q2 — Rebase-cascade policy on blocker feedback churn

**Decision: merger-driven hybrid.** When the upstream PR is
force-pushed, the merger (which has its own dedicated worktree and
no claim conflicts on the child) attempts a `git rebase` of each
stacked child onto the new upstream tip:

- Clean rebase → `git push --force-with-lease`, comment on the child
  PR, leave reviewer / verdict labels intact (the rebase preserves
  approval unless the upstream change actually re-touches the
  child's commits).
- Conflict → `git rebase --abort`, add `fleet:needs-base-update` to
  the child PR, post comment explaining the conflict files. The
  child's *original author* picks it up via the existing feedback
  loop (priority-3 tier in worker step 1).

This keeps the child author out of cross-worktree gymnastics and
reuses the merger's existing `fleet:semantic-conflict` /
`fleet:merger-cooldown` machinery. We do not auto-rebase from the
blocker's worktree (the blocker's author has no business touching
someone else's branch).

### Q3 — Multi-blocker tasks

**Decision: single-blocker only for v1.** A task with
`Blocked by: T-A, T-B, T-C` is NOT eligible for stackable claim
even if all three blockers have open PRs. Picking ONE base is a
design call (which to prefer? what about transitive conflicts?) and
multi-blocker stacking would require either cherry-picking or
merge-style integration that the merger does not have machinery for.

Multi-blocker tasks remain in their current state (skipped until
all blockers merge). v2 can revisit if the data shows multi-blocker
chains are a hot bottleneck.

## Approach

Six PRs. PRs 1–3 form the core scheduler change; PR 4 is the merger
half of the rebase-cascade; PR 5 is reviewer awareness; PR 6 is
documentation. Order matters: PR 1 must land before PR 3 (the role
doc references `fleet-claim claim --stackable-on`), and PR 2 must
land before PR 3 (the projection field is the input to the
fall-through filter). PRs 4 and 5 can ship in parallel with each
other. PR 6 lands last after the behavior is observable.

### PR 1 — `fleet-claim`: stackable-on claim mode + helpers

`scripts/fleet/fleet-claim`:

- New subcommand
  `find-stackable-blockers <task-id>` — given a task ID, parse its
  `Blocked by:` from `origin/master:TASKS.md`. If exactly one T-NNN
  blocker AND that blocker has an open PR (queried via `gh pr list
  --head claude/<blocker-branch>` matched against the blocker's
  `Owner:` field), print the PR's URL + head ref name. Otherwise
  print nothing (exit 0). Used by scout (PR 2) and as a manual
  diagnostic.
- `claim` accepts `--stackable-on <pr-url>`. When passed, the
  blocker check (today's `check_blockers` reject) is bypassed and
  the claim records the PR's head ref name as the agent's
  stackable-base hint in a sidecar file
  `$CLAIMS_DIR/<slug>.meta` containing one line:
  `stackable_base_branch=<branch>`. Atomicity unchanged
  (mkdir-claim is still the lock).
- New subcommand
  `claim-base <task-id>` — return `master` for a normal claim, or
  the recorded branch for a stackable-on claim. Used by
  `commit-and-push` at PR-open time.
- `release` cleans up the sidecar `.meta` file if present.

No changes to the molecule subcommands — those are intra-author
stacking and remain orthogonal.

### PR 2 — Scout: pre-compute `stackable_blocker_pr` per blocked task

`scripts/fleet/fleet-state-scout`:

- For each task in `tasks_open[]` whose `blocked_by` field matches
  exactly `T-NNN` (regex `^T-\d{3}$` after trimming whitespace —
  reject multi-blocker, free-text, and PR-URL forms):
  - Look up the blocker's open PR by cross-referencing
    `repos.<repo>.prs[]`: any PR whose `headRefName` starts with
    `claude/T-NNN-`.
  - If exactly one open PR matches, attach a new field
    `stackable_blocker_pr` to the task with shape
    `{ "number": N, "headRefName": "claude/T-NNN-…", "author": "..." }`.
  - If zero or more-than-one open PRs match, leave the field off.
- Same logic on engine and game `tasks_open[]` slices, but the
  fall-through is **engine-only** for v1 (the merger doesn't
  process game PRs, so the cascade half wouldn't work). Document
  this in a top-of-loop comment in the scout. The field can still
  be populated for game tasks for visibility, but worker pickup
  (PR 3) only honors it for engine.

The scout's output (and therefore the projection) is the only
input downstream code needs — workers do not re-query gh.

### PR 3 — Role docs: workers consider stackable-blocked tasks as fallback

`/Users/evinjkill/src/IrredenEngine/.claude/commands/role-sonnet-author.md`
and `role-opus-worker.md`:

- Update step 3 ("Pick the next task") in both files. After the
  existing "no unblocked tasks" exit, add a fallback tier:
  > **Fallback: stackable-blocked tasks (engine only, v1).** If
  > the unblocked list is empty, look in `tasks_open[]` for entries
  > with `owner == "free"`, repo `engine`, model matches your
  > tier, AND `stackable_blocker_pr != null`. Pick the oldest by
  > task ID. Claim with
  > `fleet-claim claim "<task-id>" <agent> --stackable-on <pr-url>`.
- Update step 4 ("Open `fleet:wip` PR"). After branching, if a
  stackable claim is in effect (test by
  `fleet-claim claim-base <task-id>` returning a non-master ref),
  branch off the recorded base instead of master:
  ```
  base=$(fleet-claim claim-base "<task-id>")
  git fetch origin "$base"
  git checkout -b claude/<task-id>-<short-topic> "origin/$base"
  ```
  At PR-open time, pass `--base "$base"` and add the
  `fleet:stacked` label. (`commit-and-push` already does this in
  fleet-stack mode; we extend the trigger to "stackable claim"
  too.)
- Note in the role doc that game-side stackable-blocked is
  **deferred to v2** — workers should still skip it for now.

### PR 4 — Merger: cascade rebase on upstream force-push

`/Users/evinjkill/src/IrredenEngine/.claude/commands/role-merger.md`
and (if needed) `scripts/fleet/fleet-claim` for any helper:

- New step in the merger loop (between today's "stacked PR"
  reconciliation and the rebase pass): for each open PR with
  `fleet:stacked` AND base PR is OPEN, compare the cached
  `baseRefName` head SHA against the merger's last-seen SHA stored
  per-PR in `~/.fleet/merger-state/stacked-bases/<child-PR>.json`.
  If the upstream tip has moved since last iteration:
  - Check out the child PR.
  - `git fetch origin <baseRefName>`
  - `git rebase origin/<baseRefName>`.
  - Clean: `git push --force-with-lease`, comment "Auto-rebased
    onto new upstream tip <new-sha>. — fleet merger". Update
    sidecar.
  - Conflict: `git rebase --abort`. Add
    `fleet:needs-base-update` to the child PR. Comment naming
    conflict files. Update sidecar so we don't re-attempt next
    iteration. (Today's `fleet:semantic-conflict` is mechanically
    similar but routes to opus-worker; `fleet:needs-base-update`
    is its own label so the original child author owns it and so
    the queue-manager can distinguish "child PR is stale because
    upstream moved" from "child PR has internal conflicts with
    master".)
- Reset to scratch branch after each cascade attempt (same pattern
  as today's step 1c).
- One PR per iteration to bound work — same discipline as
  semantic-conflict resolution.

### PR 5 — Reviewer: cross-author stacked-PR awareness

`/Users/evinjkill/src/IrredenEngine/.claude/commands/role-sonnet-reviewer.md`
(no opus-reviewer change — opus only reviews flagged PRs and
doesn't drive stack decisions):

- The existing `fleet:awaiting-upstream-review` gating already
  works (an unapproved upstream blocks child review). Add one
  paragraph in the review-body template: when reviewing a
  `fleet:stacked` PR whose upstream is open and approved, prepend
  the verdict body with a "Stacked on #<U> (cross-author: <agent>
  on T-X). Reviewing the child diff only — upstream is approved
  separately." note. This makes the cross-author topology
  explicit so the human merger sees it without spelunking.
- Also add `fleet:needs-base-update` to the verdict-clear list so
  a child's review verdict resets when the merger applies the
  label.

### PR 6 — Documentation

`docs/agents/FLEET.md`:

- New subsection under "Stacked PRs": **Cross-author stacking
  (scheduler)**. Walks through the scenario:
  1. Worker A claims T-X, opens PR #100 with `fleet:wip`.
  2. Worker B's iteration: no unblocked tasks. Fallback tier finds
     T-Y (`Blocked by: T-X`, `stackable_blocker_pr = #100`).
     Claims with `--stackable-on`.
  3. Worker B branches off `claude/T-X-…`, opens PR #101 with
     `--base claude/T-X-…`, adds `fleet:stacked`.
  4. Reviewer gates #101 on #100's approval state.
  5. Worker A pushes a feedback fix to #100. Merger detects
     upstream tip moved, rebases #101 cleanly, force-pushes.
  6. PR #100 merges. Merger re-targets #101's base to master
     (existing logic, unchanged).
- Pointer to the design decisions above (Q1/Q2/Q3) for context.
- Note v1 limitations (engine only, single-blocker only).

## Acceptance criteria

- **A1 — Scheduler.** A worker iteration with zero unblocked tasks
  but at least one stackable-blocked task picks up the stackable
  task and opens a `fleet:stacked` PR with `--base` set to the
  blocker's branch.
- **A2 — Atomicity.** Two workers in the same iteration cannot
  both claim the same stackable-blocked task — `fleet-claim`'s
  mkdir lock continues to enforce single-claim.
- **A3 — Multi-blocker rejection.** A task with `Blocked by: T-A,
  T-B` is never picked up by the fallback tier, even if both
  blockers have open PRs. (Test: stage such a task in TASKS.md
  test fixture; scout produces no `stackable_blocker_pr` field.)
- **A4 — Cascade clean.** When upstream PR force-pushes a clean
  diff, the merger rebases the child within one iteration and
  posts a confirmation comment. Reviewer's prior approval label
  is preserved.
- **A5 — Cascade conflict.** When upstream's force-push conflicts
  with the child, the merger labels `fleet:needs-base-update`,
  posts a conflict-files comment, and the child's original author
  (the worker that opened the child PR) picks it up via the
  existing feedback loop next iteration.
- **A6 — Upstream merge re-target.** When the upstream PR merges,
  the existing `fleet:awaiting-base` re-target logic flips the
  child's base to master and removes `fleet:stacked`. (No new
  code; we just confirm the existing path still works for
  cross-author chains.)
- **A7 — Game-side opt-out.** Game tasks with
  `stackable_blocker_pr` populated are NOT picked up by workers in
  v1. (Worker step 3 fallback explicitly checks `repo == "engine"`.)

## Gotchas

- **Race: blocker merges between scout snapshot and worker
  claim.** The scout's `stackable_blocker_pr` could go stale within
  ~30s. Mitigation: at claim time, `fleet-claim claim
  --stackable-on <pr-url>` re-fetches `gh pr view <url> --json
  state`. If `state == MERGED`, drop the `--stackable-on`
  semantics and re-run a normal `claim` (blocker is now resolved,
  base = master). If `state == OPEN`, proceed. If `state ==
  CLOSED`, refuse the claim — the stack base is gone.
- **Blocker branch fetched in caller's worktree.** Cross-author
  means worker B's worktree has never seen `claude/T-X-…`. The
  step-4 doc instructs `git fetch origin <base>` before checkout
  so this is always explicit.
- **Force-push by blocker mid-merger-rebase.** Standard
  `--force-with-lease` story: lease check fails → merger labels
  `fleet:merger-cooldown`, retries next tick. No new mechanics.
- **`fleet:stacked` already in use for fleet-stack (intra-author).**
  Both modes set the label; the merger and reviewer don't need to
  distinguish — the topology is the same (PR has non-master base).
  Only the *origin* of the claim differs, which lives in the
  sidecar `.meta` file. Don't introduce a `fleet:cross-author-
  stacked` second label; we'd just be doubling the label-state
  surface for no behavioral gain.
- **Sidecar `.meta` file persistence.** `~/.fleet/claims/` is host-
  local; if a worker iteration crashes, the sidecar may outlive
  the claim. `release` (and `release-stack`) must clean it up.
  Same recovery path as today's claim file orphans.
- **Plan-doc bookkeeping.** When the scheduler picks up a
  stackable-blocked task, the plan file (`.fleet/plans/T-Y.md`)
  was written assuming T-Y starts on master. The plan body may
  reference "after T-X lands". Rewriting the plan is out of scope;
  the worker reads it as-is and applies its judgment, which it
  already does for plans whose blocker landed before pickup.

## Out of scope (v2)

- Multi-blocker stacking. Open question: cherry-pick from
  secondary blockers, or merge them in? Both paths add reviewer
  surface.
- Game-side scheduler stacking. Game has no merger today;
  cross-author cascade has no actor.
- Cross-repo stacks (engine T-X, game T-Y blocked-by-T-X). Not a
  real configuration today.
- Auto-rebase from the upstream-author's worktree. Worktree
  ownership invariants make this messy; merger already covers it.
