## Architectural direction (locked in after v1 design pass)

**Sprites are screen-composite layer, not trixel content.** A sprite
is a textured quad drawn at the same pipeline stage as
`FRAMEBUFFER_TO_SCREEN`. Sprites skip the voxelâ†’trixelâ†’framebuffer
machinery entirely. The existing `SPRITE_TO_SCREEN` enum slot is the
right home; the existing `system_sprites_to_screen.hpp` stub (which is
a copy-paste of `system_framebuffer_to_screen.hpp`) gets replaced.

**Flat-depth semantics.** Each sprite is a 2D quad, so within a single
sprite every pixel shares one iso-projected depth (derived from the
anchor's `C_Position3D`). Sort key between sprites is a per-entity
scalar, not a per-pixel depth-buffer problem.

**v1 depth model.** Sprites use `C_Position3D`. Depth =
iso-projection of the anchor point, computed CPU-side each frame for
the sort. No separate depth component in v1.

**Anchor.** `C_Sprite` has `vec2 anchor_` in local [0,1] UV space,
default `{0.5, 0.0}` = bottom-center. Screen position of the quad's
origin corner = `isoProject(C_Position3D.pos_) - anchor_ * spriteSize`.

**Z-sort scope in v1.** Sort sprites against each other by iso depth.
Do NOT z-sort sprites against canvas framebuffers â€” canvases keep
their current fixed registration order (main / background / gui).
Sprites draw on top of `main`, below `gui`. Cross-layer sort is a
Part 2 concern.

**Non-iso future path.** Design note in child 1 documents the seam: a
later `C_Depth` component + `C_Position2D` can replace the
iso-projection call with a direct depth value. Sort key is `float`
either way. v1 keeps the projection call replaceable but doesn't
implement the alternate path.

**Drawing.** One instanced draw per frame. CPU sorts sprite entities
by iso depth, builds an instance buffer (per-instance: `mat4`
transform, UV rect, texture index, tint). Issues
`drawArraysInstanced` against the shared quad VAO
(`QuadVAOArrays`) with a new `SpritesToScreenProgram` fragment
shader that samples the sprite atlas/array.

### Child 1 â€” Sprite data model + components `[opus]`

Area: `engine/prefabs/irreden/render/`, docs.

- Write a design note (location TBD â€” `engine/render/CLAUDE.md`

### Child 3 â€” `SPRITES_TO_SCREEN` render pass `[opus]`

Area: `engine/prefabs/irreden/render/systems/`,
`engine/render/src/shaders/{glsl,metal}/`.

- Replace `system_sprites_to_screen.hpp` stub with a real system
  registered at the `FRAMEBUFFER_TO_SCREEN` pipeline stage. Runs
  immediately after the main canvas's `FRAMEBUFFER_TO_SCREEN` tick,
  before the GUI canvas's tick.
- CPU-side: iterate entities with `C_Sprite` + `C_Position3D`,
  compute iso-projected depth of each sprite's anchor, sort by
  depth back-to-front, build a per-instance SSBO / mapped buffer
  containing `{ mat4 transform, vec4 uvRect, uint textureIndex,
  vec4 tint }` per sprite.
- GPU-side: single `drawArraysInstanced` against the shared
  `QuadVAOArrays` (6-vert unit quad), new `SpritesToScreenProgram`
  fragment shader samples the sprite texture at `uvRect`.
- Texture feed: v1 uses a single bound `C_SpriteSheet` atlas (one
  sheet per `drawArraysInstanced` call, or a bound texture array
  with per-instance index â€” pick whichever is cleaner given the
  engine's existing texture API). Document the choice in the
  design note.
- Ship GLSL + MSL shaders in lockstep. Follow
  `.claude/skills/backend-parity/SKILL.md`.
- Do NOT fold sprites into `FRAMEBUFFER_TO_SCREEN`. They are a
  separate system at the same pipeline stage.
- Do NOT implement sprite-vs-canvas z-sort â€” explicitly Part 2.

Acceptance:
1. A single sprite at `C_Position3D = (0, 0, 0)` renders at screen
   center with bottom-center anchor, pixel-perfect scaling at
   zoom 1.
2. Multiple sprites composite correctly â€” back-to-front by iso
   depth, no z-fighting, correct alpha compositing.
3. 50+ sprites in a single frame cost one draw call, not N.
4. Builds clean on `linux-debug` AND `macos-debug`. Visual parity
   between backends.
5. `render-debug-loop` screenshot committed to
   `docs/pr-screenshots/`.
6. `fleet-run IRShapeDebug` no regressions on existing shots.

Depends on: child 2.

### Child 4 â€” `C_SpriteAnimation` + animation system `[opus]`

Area: `engine/prefabs/irreden/render/`.

- `C_SpriteAnimation` component:
  - Current sub-animation name (ref into `C_SpriteSheet`'s map).
  - Current frame index within that sub-animation.
  - Elapsed-in-frame seconds.
  - Loop mode: `ONCE` / `LOOP` / `PING_PONG`.
  - Playback speed multiplier (default 1.0).
- UPDATE-phase system:
  - Per-tick `deltaTime(UPDATE)` advance.
  - Flips frames at the configured FPS from the sheet's named
    animation entry (not render rate).
  - Writes the resolved frame's UV rect back onto `C_Sprite.uvRect`.
- API surface lives in a prefab-scoped namespace per the layering
  principle in #266 â€” NOT in `IRRender::`. Suggested:
  `IRPrefab::Sprite::playAnimation(entityId, "walk_left", mode, speed)`,
  `stopAnimation`, `getCurrentFrame`.

Acceptance:
1. Animation cycles at the sheet-specified FPS, independent of
   render rate.
2. `LOOP` wraps, `ONCE` stops on final frame, `PING_PONG` reverses
   at endpoints.
3. Named sub-animation switch mid-playback works without a frame
   glitch (resets `elapsed` + `frameIndex` to the new
   sub-animation's first frame).
4. Unit test for the time-advance math (no render dependency).
5. Builds clean on active preset.

Depends on: child 1. Soft dep on child 3 for visual confirmation;
the math test does not need child 3.

## Out of scope (follow-ups if anyone asks)
