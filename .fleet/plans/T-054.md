# Plan: #310 вҖ” World Z-yaw rotation across the trixel/SDF pipeline

## Goal

Add support for **continuous Z-yaw rotation of the world view** while
preserving the engine's signature O(1) trixel rasterization
optimization. Continuous-rotation game logic вҖ” pseudo-shooter style
where the line of sight is fixed in screen space and the world
rotates around the player вҖ” must work end-to-end (movement, aim,
picking, hit detection).

## The constraint that drives the architecture

The trixel rasterizer's signature optimization is in
`engine/render/src/shaders/c_voxel_to_trixel_stage_1.glsl:71-76` and
`engine/render/src/shaders/ir_iso_common.glsl:9-18`:

```glsl
ivec2 pos3DtoPos2DIso(ivec3 position) {
    return ivec2(
        -position.x + position.y,
        -position.x - position.y + 2 * position.z
    );
}

// in stage_1:
const ivec2 canvasPixel =
    trixelCanvasOffsetZ1 + ivec2(floor(frameCanvasOffset))
    + ivec2(gl_LocalInvocationID.xy)
    + pos3DtoPos2DIso(voxelPositionInt);
writeDistanceTap(canvasPixel, voxelDistance);
```

Each voxel writes to **exactly two known integer trixel pixels per
visible face** via `imageAtomicMin`. Cost is `O(visible_voxels Г— 6)`
per frame, not `O(canvas_pixels Г— visible_voxels)`. This works ONLY
because the iso projection maps integer voxel positions to integer
trixel positions вҖ” no fractional sampling, no antialiasing in the
hot path.

### Mathematical fact

Composing Z-rotation `R(Оё)` with the iso projection `M`:

```
M = [-1  1  0]      MВ·R(Оё) = [-cos Оё - sin Оё,  -sin Оё + cos Оё,  0]
    [-1 -1  2]               [-cos Оё + sin Оё,  -sin Оё - cos Оё,  2]
```

Integer-input вҶ’ integer-output requires `sin Оё вҲҲ в„Ө` and `cos Оё вҲҲ в„Ө`,
plus `sinВІ + cosВІ = 1`. The only solutions are
**Оё вҲҲ {0В°, 90В°, 180В°, 270В°}**. Any other yaw value puts voxels at
fractional trixel positions and breaks the integer-rasterization
optimization.

## Architectural decision: two-yaw model (Option B)

Continuous visual rotation, cardinal-snap rasterization, residual
2D rotation in a screen-space composite pass.

### Data flow

```
C_CameraYaw { float visualYaw_; }        вҶҗ gameplay writes this
                вҶ“
        (per-frame derive in render pipeline)
                вҶ“
    в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ

    = round(visual / 90В°)    = visualYaw - rasterYaw
      Г— 90В°                    вҲҲ [-45В°, 45В°)
    вҶ“                         вҶ“
    Voxel raster shader       Screen-space 2D rotate pass
    (4 cardinal cases)        (continuous angle)
                              вҶ“

```

- **`visualYaw_` is the only canonical value.** Game logic, picking,
  movement, aim вҖ” all read this. The cardinal split is an internal
  renderer concern.
- **`rasterYaw` and `residualYaw` are derived per frame** by a small
  helper in the foundation task (#317). Not stored as state; pure
  function of `visualYaw_`.

### Per-frame composition (both directions)

Forward (visible result):
```
screen_pixel = (residual 2D rotate)( cardinal_iso_raster( world ) )
             = M В· R(rasterYaw) В· world  вҲҳ  R2D(residualYaw)
             = (visible orientation = visualYaw)
```

Inverse (picking):
```
world = R(-rasterYaw) В· MвҒ»В№ В· R2D(-residualYaw)( screen_pixel )
```

Both compose to identity at `residualYaw=0` (cardinal yaw values
exactly), preserving yaw-zero parity for existing creations.

### Gameplay API surface

```cpp
// in engine/prefabs/irreden/render/ or wherever camera lives
namespace IRPrefab::Camera {
    void  setYaw(float radians);
    float getYaw();              // continuous; what gameplay reads
}
```

For the pseudo-shooter case:
```cpp
float yaw      = IRPrefab::Camera::getYaw();
vec2  aimDir   = rotate2D({0, 1}, yaw);  // screen-bottom-to-top in world coords
auto  pickedPos = pickWorld(screenBottom, screenTop);  // exact under any yaw
```

Snap is invisible to gameplay code.

## Known artifacts at 45В° boundaries

Both transient, both during continuous rotation only, both acceptable
for v1.

### 1. Sub-pixel raster discrepancy

Voxel `(3, 5, 0)` rasterized at `rasterYaw=0В°` lands at one integer
trixel; the same voxel rasterized at `rasterYaw=90В°` (under `(5, -3,
0)`) lands at a slightly different integer trixel. As `visualYaw`
crosses 45В°, the canvas content "swaps" between cardinal raster
yaws вҖ” producing a 1-pixel discrepancy at the swap point.

**Mitigation:** the screen-space residual pass uses bilinear
filtering. The 1-pixel discrepancy is hidden as a momentary
softness during transitions. At rest (residual=0 at cardinal
angles), pixel-perfect quality is restored.

### 2. Procedural face brightness flip

`adjustColorForFace` (in `ir_iso_common.glsl`) gives Z=1.25, X=1.0,
Y=0.75 brightness based on the face index. At cardinal swaps, the
face index вҶ’ world-direction mapping rotates discontinuously, so a
wall's apparent shading flips at 45В° boundaries.

**Mitigation paths:**
- The sun-shadow lighting pipeline (T-013, merged) is direction-
  driven and immune to this.
- Workaround in fallback procedural path: derive face brightness
  from the WORLD face direction (after raster yaw inverse), not the
  raster-frame face index. Cheap fragment-shader change, listed as
  follow-up if visible.
- For v1, document the artifact in the design note. Not a blocker.

## Out of scope (likely follow-up epic)

- **Lighting / shadow math under non-cardinal yaw.** Sun-shadow
  pipeline already direction-aware so it should work; flag during
  #317 if anything assumes yaw=0.
- **Pitch / roll** (rotation around X or Y world axes).
- **Continuous voxel rasterization (Option C вҖ” true any-angle world
  perspectives).** If the project ever needs to inspect asymmetric
  structures from arbitrary angles (45В°-from-cardinal showing a
  different face), this requires fractional trixel rasterization
  which loses the signature O(1) optimization. File as v2 epic.
- **Procedural face brightness fix** (artifact #2 above).

## Decomposition вҖ” 5 children (stack)

| # | Issue | Notes |
|---|-------|-------|
| **#317** | view/camera Z-yaw foundation | Adds `C_CameraYaw { visualYaw_ }`, per-tick helper deriving `(rasterYaw, residualYaw)`, threads both to GPU feeders. Foundation. |
| **#311** | trixel rasterization (cardinal-snap, re-scoped) | Shader picks one of 4 basis-vector permutations from `rasterYaw`. Title and scope updated from "arbitrary" to "cardinal-snap." Performance: any cardinal yaw equals yaw=0 cost. |
| **#312** | SDF rasterization | Stays continuous вҖ” SDF math has no integer-alignment requirement. Reads `visualYaw` directly. The free half. |
| **NEW** | screen-space 2D residual yaw composite pass | NEW. Slots between `TRIXEL_TO_TRIXEL` and `FRAMEBUFFER_TO_SCREEN`. Reads `residualYaw`, rotates the trixel canvas by that angle into the framebuffer with bilinear filtering. GLSL + MSL. The continuity provider. |
| **#313** | screenвҶ’world picking | Inverse composes `(-residualYaw)` then `(-rasterYaw)`. Sequenced last; inverts a stable composition. |

### Stack order


#311 вҶ’ NEW is the critical path: the residual pass must know the
canvas was rasterized at the cardinal yaw it expects. Once both land,
picking can compose the full inverse.

## Validation plan

Each child has its own acceptance; epic-level acceptance per #310:

- [ ] Visual + perf parity at `visualYaw = 0` (no regression on any
      existing creation).
- [ ] Trixel and SDF shapes render correctly at arbitrary yaw,
      including animated yaw frame-to-frame.
- [ ] ScreenвҶ”world picking returns correct world coords under
      arbitrary yaw (verify with mouse-pick at every 15В° increment
      across 360В°).
- [ ] Backend parity: identical behavior on OpenGL and Metal at all
      cardinal yaw values + at least 4 non-cardinal angles.
- [ ] Per-frame regression at yaw=0 within a small (single-digit %)
      bound; cost at arbitrary yaw characterized in PR bodies.
