# T-091 / Issue #428 — Render: AO via trixelDistances (drop OccupancyGridBuffer)

Derived from the T-09X follow-up section of `~/.fleet/plans/issue-358.md` (architect direction
filed alongside T-071 / PR #423).

## Goal

Migrate `c_compute_voxel_ao.{glsl,metal}` off the `OccupancyGridBuffer` SSBO (slot 28).
Read 4 face-tangent neighbour pixels from `trixelDistances` instead of calling
`occupancyGetBit` against the CPU-built world-space bitfield. Drops AO's dependency
on the occupancy grid; unblocks the final teardown (T-092).

## Current algorithm

`c_compute_voxel_ao.glsl:131-167`: for each visible surface pixel, reconstruct
`surfaceVoxel`, then sample 4 face-outward + ±tangent neighbour cells from
`OccupancyGridBuffer`. Each filled neighbour darkens by 10%.

## Target algorithm

For each visible surface pixel:
1. Reconstruct `pos3D` and `face` as today (`isoPixelToPos3D`, `face = encoded & 3`).
2. Sample 4 neighbour iso pixels in `trixelDistances` along the face's two tangent axes
   (in iso-space — project world-space tangents through the iso transform once on the CPU
   and ship in the existing `FrameDataVoxelToTrixel` UBO, or compute inline).
3. For each sampled neighbour: decode its `pos3D'` from its `rawDepth`. If `pos3D'` is
   closer to the iso camera than the receiver's plane along face-outward (i.e. the
   neighbour pixel is in front of the receiver's face plane by less than ~1 voxel),
   count as occluding.
4. AO factor as today: `1 - 0.10 * occl`.

Same crease-darkening behaviour; no occupancy SSBO read; no CPU 2 MB per-frame upload
contributing to the AO path. Cost stays O(canvasPixels) with 4 texel reads per pixel.

## Open questions for the implementing agent

1. **Iso-axis-aligned occluder occluded by receiver** — does it matter visually? An
   occluder behind the receiver in iso depth is hidden anyway, so its absence from
   `trixelDistances` is exactly correct for AO purposes. Validate via `render-debug-loop`
   against existing AO screenshots; acceptable diff is bias-only at crease edges.
2. **Sub-voxel resolution at high zoom / subdivision modes** — current AO converts `pos3D`
   back to integer voxels via `roundHalfUp`. Screen-space sampling should respect
   `voxelRenderOptions.x` subdivision mode and read sub-voxel neighbours at full zoom.
   Spec the sample distance in iso pixels, not voxels.
3. **SDF shapes** — already in `trixelDistances` via `c_shapes_to_trixel`, so they
   participate in AO automatically. No separate SDF rasterization step needed. This is
   a regression-fix relative to historical AO behaviour (the SDF-into-occupancy step
   is deleted in T-071).

## Acceptance

1. `c_compute_voxel_ao.{glsl,metal}` no longer reads `OccupancyGridBuffer` / slot 28.
2. `system_compute_voxel_ao.hpp` no longer binds `OccupancyGridBuffer`.
3. AO crease-darkening within ~1 iso pixel of edge migration vs. pre-migration reference
   (captured before this PR; compared via `render-debug-loop`).
4. SDF shapes get crease AO on adjacent surfaces.
5. `fleet-build --target IRShapeDebug` clean on `linux-debug` AND `macos-debug`.
