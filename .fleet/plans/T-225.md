## Approach

### Per-worker buffers

`EntityManager` holds:

```cpp
struct WorkerStaging {
    std::vector<DeferredComponentSet> componentSets;
    std::vector<DeferredComponentRemove> componentRemoves;
    std::vector<EntityId> markedForDeletion;
    std::vector<DeferredEntityCreate> entityCreates;
};
std::vector<WorkerStaging> m_workerStaging;  // sized N+1 (main + workers)
```

Index 0 is main; indexes 1..N are workers. `IRJob::workerId()`
returns 0 from main, 1..N from workers (matches the enkiTS
convention).

No lock needed on writes ŌĆö each worker touches only its own
buffer.

### Drain

`flushStructuralChanges()`:

```cpp
IR_ASSERT_MAIN_THREAD();  // serial fence ŌĆö workers cannot drain
for (int wid = 0; wid < m_workerStaging.size(); ++wid) {
    auto& staging = m_workerStaging[wid];
    applyComponentSets(staging.componentSets);
    applyComponentRemoves(staging.componentRemoves);
    applyEntityCreates(staging.entityCreates);
    applyDeletions(staging.markedForDeletion);
    staging.clear();
}
```

Workers in `workerId` order, not iteration-order over a map.
Determinism matters for `--auto-screenshot` reproducibility.

### Entity ID allocation

`createEntity` from a worker returns a fresh ID. Use a
`std::atomic<EntityId> m_nextEntityId` counter ŌĆö NOT a per-worker
range. Range-based would balloon the archetype index when most
workers are idle (sparse ID space).

The atomic fetch_add is the only contended op in this path ŌĆö
amortized over per-row work it's cheap.

### Validator lift

In T-224's validator: delete the
`A.mutates_archetype_graph && B.mutates_archetype_graph` FATAL.
The `Spawns` / `Destroys` tags stay (they're useful for the
trait's flag set), just no longer used as a conflict source.

## Gotchas

- `createEntity` ID allocation: atomic counter, NOT per-worker
  range. The temptation is to give each worker a range to avoid
  the atomic ŌĆö DON'T. Sparse ID allocation breaks the archetype
  index assumptions.
- Drain order MUST be deterministic. Iterate workers by index
  (0..N), not by hash-map order. `--auto-screenshot`
  reproducibility breaks otherwise.
- Don't try to share buffers between same-thread callers and
  worker callers. Same-thread callers (main thread, workerId=0)
  shouldn't pay any atomic op; keep the dispatch wrapper.
- The per-worker buffer count is fixed at `JobManager::start`
  time. If the worker count changes mid-frame, things break
  spectacularly. Don't expose a runtime resize API.
