# Plan: Pipeline groups + cross-system validation (T-224)

- **Issue:** #1071
- **Model:** opus
- **Date:** 2026-05-22
- **Epic:** #226 — see `~/.fleet/plans/issue-226.md` for full context

## Scope

Phase 3 of the multithreading epic. Lifts parallelism from
per-system (T-222) to across-systems within a group. The
cross-system access validator is what turns Layer 4's type
derivation from an unused trait into a registration-time gate.

## Affected files

- `engine/system/include/irreden/ir_system.hpp` — add
  `registerPipelineGroups(IRTime::UPDATE, { {a, b}, {c}, ... })`
  public API. Existing `registerPipeline` keeps working.
- `engine/system/include/irreden/system/system_manager.hpp` — change
  the per-pipeline storage from `std::list<SystemId>` to
  `std::vector<std::vector<SystemId>>` (groups of systems).
- `engine/system/src/system_manager.cpp` (`executePipeline`) — for
  each group, dispatch all systems in the group concurrently via
  `IRJob::runGroup` or equivalent; wait for the group to
  complete before advancing to the next group. `flushStructuralChanges`
  runs between groups (preserves the existing fence cadence).
- `engine/system/src/system_manager.cpp` (`World::start` hook) —
  cross-system access-conflict check. Pseudocode in #226 Layer 4.
  FATAL on conflicts at startup, naming both systems + the
  offending component or tag.
- `engine/world/src/world.cpp` — call the validator hook after all
  systems registered, before the main loop starts.
- `engine/system/CLAUDE.md` — document the groups API + the
  cross-system validator rules. Show one example group declaration.
- Wherever the engine's UPDATE pipeline is declared today
  (probably `engine/world/src/world.cpp` or a prefab init) —
  migrate to `registerPipelineGroups`. Partition emerges from the
  validator's accept list.
- `docs/perf-reports/threading_phase3.md` — new. Speedup beyond
  T-222.

## Approach

### Validator check

For every pair `(a, b)` in the same group, FATAL if any of:

```cpp
const auto& A = systemAccess(a);
const auto& B = systemAccess(b);

if (A.main_thread_only || B.main_thread_only) FATAL(...);
if (intersects(A.writes, B.reads)) FATAL(...);
if (intersects(A.writes, B.writes)) FATAL(...);
if (intersects(B.writes, A.reads)) FATAL(...);
if (A.mutates_archetype_graph && B.mutates_archetype_graph) FATAL(...);
```

Error messages MUST name both systems + the specific conflicting
component (or the offending tag). Vague "these systems conflict"
isn't debuggable.

### Group dispatch

```cpp
for (const auto& group : pipeline.groups) {
    IR_PROFILE_BLOCK("group", COLOR_SYSTEMS);
    if (group.size() == 1) {
        executeSystem(group[0]);  // existing serial path
    } else {
        // concurrent dispatch across the group
        IRJob::runGroup(group.size(), [&](size_t i) {
            executeSystem(group[i]);
        });
    }
    flushStructuralChanges();  // existing fence
}
```

### Pipeline migration

Don't pre-guess the group partition. Land the API + validator
first; then migrate the declared pipeline by trying group
candidates and letting the validator's accept list determine the
final partition. Likely first groups based on quick survey:

- `{velocity, drag, gravity}` — all write `C_Velocity3D` reading
  from different sources. Verify with `SystemAccess` dump.
- `{globalPosition}` — reads `C_Position3D` + `C_PositionOffset3D`,
  writes `C_PositionGlobal3D`. Probably own group.
- `{lifetime}` — destroys. Probably own group until T-225.

## Acceptance criteria

- Validator rejects:
  - Hand-constructed group sharing a write column — unit-tested.
  - Group containing a `MAIN_THREAD` system — unit-tested.
  - Two spawners in same group (`mutates_archetype_graph`) —
    unit-tested. (Lifts in #1072.)
- Engine UPDATE pipeline reorganized into groups; smoke-test via
  `IRShapeDebug --auto-screenshot 60` unchanged.
- `perf_grid_matrix.sh` shows additional speedup beyond T-222's
  number on the 32K and 262K cells.

## Gotchas

- Error messages MUST name the specific conflicting component,
  not just "systems conflict". Author-side debuggability is the
  whole point of the type-derived check.
- `intersects(A.writes, B.reads)` is asymmetric — check both
  directions. Easy to write only one and ship a footgun.
- Determinism: cross-system parallel groups break per-system
  observation order. That's the user's contract when they
  declare a group — same as today's "declare unrelated systems
  in any order". Don't try to preserve order.
- Don't add `registerPipelineGroups` as a *replacement* for
  `registerPipeline`. Both coexist — single-group calls
  (`registerPipeline(UPDATE, a, b, c)`) become syntactic sugar
  for one-system groups.
