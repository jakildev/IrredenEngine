# Plan: GPU particles Phase 2 — batch CPU-side Metal spawns

- **Task:** T-159
- **Issue:** #632
- **Model:** opus
- **Date:** 2026-05-11

## Scope

Eliminate per-spawn Metal `subData` orphan churn on `C_GPUParticlePool`'s
SSBO by batching CPU-side spawn writes into a frame-scoped pending list
and flushing them once per frame as run-coalesced `subData` calls. The
common case — `reserveSlot()`'s linear-scan reclamation hands out
sequential indices — collapses to exactly 1 `subData/frame` regardless
of the per-frame spawn count.

## Approach

Approach (a) from #632, refined to avoid clobbering GPU-authored state:

1. Add `std::vector<std::uint32_t> pendingIndices_` to `C_GPUParticlePool`,
   reserved to `capacity_` in the ctor so the hot path never reallocates.
2. `writeSlot(idx, particle)` mirrors into `particles_[idx]` (for
   `reserveSlot()`'s dead-slot scan) and appends `idx` to
   `pendingIndices_`. No `subData` here.
3. New `flushPendingSpawns()` method:
   - Sort `pendingIndices_` and dedupe (the same slot can be written
     more than once per frame if reclaimed; we want one upload per slot).
   - Walk the sorted list, identify contiguous runs, issue one
     `Buffer::subData` per run using `&particles_[runStart]` as the
     source pointer. Critically: we never re-upload "untouched" slots,
     because those would clobber the GPU's per-frame position/lifetime
     updates.
   - Clear `pendingIndices_` (preserves capacity).
4. `UPDATE_GPU_PARTICLES` per-pool tick calls `pool.flushPendingSpawns()`
   before the update dispatch reads the SSBO.
5. `clear()` keeps its existing immediate full-buffer `subData` (it's a
   reset, not a hot path) and resets `pendingIndices_` to empty.
6. Extend `creations/demos/gpu_particles` with a `--spawns-per-frame N`
   CLI arg and a small RENDER-pipeline spawner system inserted ahead of
   `UPDATE_GPU_PARTICLES`. Default 0 preserves baseline behavior.

## Affected files

- `engine/prefabs/irreden/render/components/component_gpu_particle_pool.hpp` —
  add `pendingIndices_`, change `writeSlot` to append-only, add
  `flushPendingSpawns()`.
- `engine/prefabs/irreden/render/systems/system_update_gpu_particles.hpp` —
  call `pool.flushPendingSpawns()` at the top of the per-pool tick.
- `creations/demos/gpu_particles/main.cpp` — CLI arg for continuous
  emission rate + a spawner system registered ahead of
  `UPDATE_GPU_PARTICLES` when the rate is non-zero.

## Acceptance criteria

- `IRGpuParticles --spawns-per-frame N --auto-screenshot 10` at
  N ∈ {10, 100, 1000} steady-state issues exactly 1 `subData/frame`
  against the pool buffer in the typical (sequential-index) case.
- Wrap-around / sparse-reclaim cases still bound by run count, not
  by N — the typical scenario remains 1–2 `subData/frame`.
- Cross-backend visual parity (OpenGL on Linux fleet; Metal on macOS
  via the cross-host smoke label).
- `fleet-build --target IRGpuParticles` clean.
- Existing baseline screenshots for the demo unchanged (default
  `--spawns-per-frame 0` keeps seed-only behavior).

## Gotchas

- **CPU mirror is stale for position/lifetime.** The GPU update kernel
  mutates those every frame; the CPU mirror is no longer the source of
  truth between mutations. The flush MUST only `subData` slots that
  were just written this frame — never re-upload untouched contiguous
  ranges, because that would overwrite live GPU-authored state.
- **Pipeline ordering.** Spawners must run before
  `UPDATE_GPU_PARTICLES`. Existing seed in `initEntities()` runs before
  the game loop starts, so its first flush happens on frame 0 of the
  RENDER pipeline.
- **`clear()`** stays a full immediate `subData`. It's caller-initiated
  (not per-frame), and its semantics are "zero every lifetime", so the
  full upload matches intent and we don't accidentally retain pending
  indices across the reset.
- **No new shader, no new bind point.** This PR is the CPU-side batch
  only; scatter-compute is unnecessary because `reserveSlot()` already
  produces sequential indices in steady state. If a future workload
  ever produces sparse reclaim patterns at high spawn rates, the
  follow-up is a scatter SSBO + compute pass — not warranted today.
