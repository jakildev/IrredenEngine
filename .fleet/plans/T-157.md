# Plan: Migrate prefab systems to member-on-System<N> via registerSystem

- **Issue:** #580
- **Model:** sonnet (mechanical 1:1 refactors)
- **Date:** 2026-05-11

## Scope

The `registerSystem<N, Cs...>` helper plus concept-based hook detection
already landed in PR #592 (T-136). The CLAUDE.md "Per-system parameters"
section was rewritten in the same PR to make the member-on-`System<N>`
form the preferred path and document the explicit `Params +
setSystemParams` form as the escape hatch. The test surface
(`test/system/register_system_test.cpp`) covers all three tick
signatures, `beginTick` / `endTick`, member-field persistence across
ticks, and per-instance state separation.

**What's left** is the **migration** of 17 prefab-system header files
currently on the explicit `Params + setSystemParams` pattern. Each
migration is a pure 1:1 mechanical rewrite:

- The nested `Params` struct's fields become member fields on
  `System<N>` (retain the trailing-underscore naming).
- Each captured-lambda body becomes a named member function
  (`tick` / `beginTick` / `endTick` / `relationTick`).
- The `std::make_unique<Params>()` + `[p]` captures + `setSystemParams`
  scaffold becomes a single `registerSystem<N, Cs...>("Name")` call
  inside `create()`.
- Any helper constants (e.g. `kHitboxGuiIdentityYawEpsilon`) stay
  where they are — only the per-system params + lambda boilerplate
  changes.

Behavior, lifetime, and per-tick cost are identical by construction:
the helper does the same `std::make_unique<...>()` + raw-pointer
capture + `setSystemParams` handoff that the explicit pattern does by
hand, just behind one entry point.

## Approach

Five small PRs, each migrating a logical cluster. PRs are independent
(branch off `master`), **not stacked** — the files don't share state
or depend on each other at the migration level. Reviewers can land
them in any order.

All migrations are `[sonnet]`: pure 1:1 refactors of bounded surface
with no design judgment required. The helper plumbing (which would
have been `[opus]`) is already merged.

## Migration recipe (per system)

For each system file:

1. **Move Params fields to System<N> members.** The `struct Params {
   ... }` nested in `System<N>` becomes member fields directly on
   `System<N>`. Field names keep their trailing-underscore convention.
   Per-class initializers stay.

2. **Convert lambdas to member functions.** The captured-`[p]` lambdas
   in `create()` become named member functions on `System<N>`:
   - Tick lambda → `void tick(Cs&... cs)` (or
     `tick(EntityId, Cs&...)` / `tick(const Archetype&,
     std::vector<EntityId>&, std::vector<Cs>&...)` depending on the
     existing signature). The body changes `p->field_` to plain
     `field_` (implicit `this->`).
   - `beginTick` lambda → `void beginTick()`. Same swap.
   - `endTick` lambda → `void endTick()`. Same swap.
   - `relationTick` lambda → `void relationTick(RelComps&...)`. Same
     swap.

3. **Replace `create()` body.** Drop the
   `make_unique<Params>` + `Params *p = …` + lambda definitions + the
   final `setSystemParams(...)` line.

   - **Simple case** (no pre-`createSystem` Params population — e.g.
     `system_hitbox_mouse_test_gui.hpp`):

     ```cpp
     static SystemId create() {
         return registerSystem<NAME, Cs...>("Name"[, relationParams]);
     }
     ```

   - **GPU-resource case** (Params fields pre-populated with
     `IRRender::getNamedResource<…>("…")` cached pointers before the
     original `createSystem` call — applies to every PR-2 compute
     system and to `voxel_to_trixel` / `shapes_to_trixel` /
     `lighting_to_trixel` / `trixel_to_*` / `fog_to_trixel` /
     `framebuffer_to_screen` / similar in PRs 3-5):

     Keep the `IRRender::createNamedResource<…>(…)` calls in
     `create()` (they must run before `registerSystem` so the named
     resources exist when `beginTick`/`tick` first read them). After
     `registerSystem` returns, fetch the instance back via
     `getSystemParams<System<NAME>>(systemId)` and populate the
     member fields, then return:

     ```cpp
     static SystemId create() {
         IRRender::createNamedResource<ShaderProgram>("X", …);
         IRRender::createNamedResource<Buffer>("Y", …);
         // … all resource creation as before …

         SystemId systemId = registerSystem<NAME, Cs...>("Name"[, relationParams]);
         auto *p = getSystemParams<System<NAME>>(systemId);
         p->compactProgram_ = IRRender::getNamedResource<ShaderProgram>("X");
         p->frameDataBuf_   = IRRender::getNamedResource<Buffer>("Y");
         // … one assignment per pre-populated field …
         return systemId;
     }
     ```

     Same lifetime, same per-tick cost (one captured pointer into
     the lambdas the helper builds, identical to the explicit
     pattern). The two-step (allocate via `registerSystem`, populate
     via `getSystemParams`) is the documented seam — see
     `engine/system/CLAUDE.md` "Either path: read the params back
     from outside the tick."

   When the existing system passes a `RelationParams<...>` argument,
   forward it through `registerSystem`'s second parameter.

4. **Constants / helper namespaces stay put.** Anything outside the
   `Params` struct + lambda bodies (file-scope `constexpr`,
   helper free functions, includes) is untouched.

## Gotchas

- **`relationTick` requires `RelationParams<...>` to match the
  member's signature.** If a system today has a relation tick, pass
  the same `RelationParams<RelComps...>{}` argument to
  `registerSystem`; the helper's concept-detection on
  `relationTick(RelComps&...)` then picks it up. If you forget the
  argument, the relation-tick member won't be wired and the
  relation-walk silently no-ops. The compile error story for this
  case is weak — visually inspect after migration.
- **`tick` signature must exactly match one of the three forms.**
  `tick(C_Foo&)` and `tick(const C_Foo&)` are different signatures
  to the concept; mirror the existing lambda's parameter types
  (const-ness, value-vs-reference) exactly. A signature mismatch
  produces a `static_assert` in
  `MakeMemberTickFn::build` ("System<N> must declare a tick(...)
  member matching one of the three valid signatures").
- **`voxel_to_trixel.hpp` has two specializations in one file.**
  Migrate both in the same PR. They share no state but share
  the file, so a half-migrated file is a non-starter.
- **Pre-populated Params fields require the two-step `create()`
  shape** (see Migration recipe step 3). The simple
  `return registerSystem<…>(…);` form only works for systems whose
  Params are filled lazily in `beginTick` (e.g. `hitbox_mouse_test_gui`).
  Every compute system in PR 2, plus `voxel_to_trixel`,
  `shapes_to_trixel`, `lighting_to_trixel`, `trixel_to_trixel`,
  `trixel_to_framebuffer`, `framebuffer_to_screen`,
  `fog_to_trixel`, `text_to_trixel`, `sprites_to_screen`, and
  `debug_culling_minimap` cache `getNamedResource<…>` pointers in
  `create()` and so need the
  `getSystemParams<System<N>>(systemId)` populate step. Skipping
  the populate leaves the pointer members null and the first tick
  crashes on a null shader / buffer. Visually inspect every
  migrated `create()` for `p->fieldName_ = …` lines in the
  pre-PR diff and make sure each has a paired
  `instance->fieldName_ = …` after `registerSystem`.
- **Implicit `this->` capture vs. explicit `p->`.** Inside member
  functions you write `field_` (not `p->field_` or `this->field_`).
  Don't accidentally leave a stale `p->` reference from copy-paste.
- **`SystemName` enum must already contain the system's enum value.**
  These migrations don't touch the enum since the systems are
  already registered with valid `SystemName`s. Skip the
  "new-system" checklist item.
- **No new files, no new components, no new public API.** The
  surface is `ir_system.hpp` + the 17 prefab headers. Anything
  beyond that should bounce back as a scope question.
- **One PR per cluster, not per file.** PR 1 is the canonical-
  pattern proving PR with one file; PRs 2-5 cluster by area to
  keep the PR count manageable. A single-file PR for each of 17
  files would swamp the review queue.
