# Plan: Camera-anchored light volume (rescoped from #360)

- **Issue:** #360
- **Model:** opus
- **Date:** 2026-05-02

## Scope

Re-scope from the original two-grid proposal to **light-volume only**.
The 256³ occupancy-grid portion is moot — T-071 / T-091 / T-092 delete
the occupancy grid entirely (sun shadow already moved off it; AO and
light-volume migrate, then the grid disappears).

After T-072 (GPU light-volume propagation, jump flooding) lands, the
128³ RGB light volume is still anchored at world origin with fixed
extent. This task makes the volume's origin track the iso camera so:

- Lighting fidelity is preserved when creations pan far from origin
  (today, anything past `±64` voxels is silently clamped).
- Per-frame coverage shifts to the visible region without growing the
  texture memory footprint.

**Blocked by:** T-072. Camera-anchoring layers on top of the GPU
jump-flood implementation; building it on the CPU producer would be
discarded work.

## Approach

After T-072 introduces the seed + propagate compute passes, add a
`worldOrigin_` field to the light-volume frame data and update both
producer and consumer to translate world coordinates into local-grid
coordinates before sampling.

1. **Compute the volume origin.** In the system that drives the GPU
   light-volume passes, compute each frame:
   - `cameraCenter` = the iso camera's world-space target. Use the
     same call site that drives the trixel canvas centre — NOT the
     GLFW window centre. The two diverge under non-default zoom and
     viewport offset.
   - Snap `cameraCenter` to a multiple of the grid cell size
     (`floor(cameraCenter / kCellSize) * kCellSize`) to avoid
     sub-cell sliding artifacts. Voxel-aligned snap is fine for v1;
     a finer snap (or hysteresis around the snap boundary) can be
     revisited if shimmer appears at high zoom.
   - Persist the origin in the existing light-volume frame-data
     struct that ships uniforms to the GPU each frame.

2. **Producer (seed pass).** The seed compute shader writes
   `C_LightSource.emissiveRgb_` into volume texels at
   `(lightWorldPos - worldOrigin_) / kCellSize`. Seeds outside
   `[0, 128)` after translation are skipped — no clamp, no wrap.
   Out-of-volume lights contribute nothing this frame, by design.

3. **Propagate pass(es).** Jump flooding is purely local — neighbour
   reads/writes use grid-local coordinates already. The propagate
   shader does **not** need to know `worldOrigin_`. Don't pipe it in;
   unused uniforms become a maintenance trap.

4. **Consumer (`c_lighting_to_trixel.glsl` + `.metal`).** Subtract
   `worldOrigin_` before computing the volume-sample texel. This is
   the existing `texelFetch` site against the volume texture; the
   change is one extra UBO field plus one vec3 subtract.

5. **Origin-shift handling.** When `worldOrigin_` changes between
   frames (camera panned past one grid-cell boundary), the previous
   frame's volume contents are stale relative to the new origin.
   Treat the shift as a full invalidation: clear the volume to zero
   on shift frames before the seed pass writes. Cheap — seed +
   propagate is already a fixed handful of dispatches per frame.
   If T-072 ships ping-pong volume textures, both buffers must be
   cleared on shift frames or the next read-from-old buffer leaks
   stale lighting.

6. **CLAUDE.md updates.** `engine/render/CLAUDE.md` and
   `engine/prefabs/irreden/render/CLAUDE.md` light-volume sections
   should note that the volume is camera-anchored with a snapped
   world origin, and that lights outside the volume are skipped at
   seed time.

## Affected files

- `engine/prefabs/irreden/render/systems/system_compute_light_volume.hpp`
  — origin computation, uniform-struct population. (After T-072 this
  owns the GPU dispatch + seed-data marshalling, not the CPU BFS.)
- The light-volume frame-data UBO struct introduced by T-072 — add
  `worldOrigin_` (vec3 / vec4 with padding per platform's std140
  rules).
- `engine/render/src/shaders/glsl/c_compute_light_volume_seed.glsl`
  (+ `.metal` counterpart) — read origin uniform, translate write
  index, skip out-of-volume seeds.
- `engine/render/src/shaders/glsl/c_lighting_to_trixel.glsl` (+
  `.metal`) — subtract origin before the volume-sample texel
  calculation.
- `engine/render/CLAUDE.md`, `engine/prefabs/irreden/render/CLAUDE.md`
  — mention camera-anchored volume + snap rule.

## Acceptance criteria

1. `IRLightingCombined`, `IRLightingEmissive`, and any other lighting
   demo with a non-trivial camera offset render with full fidelity
   when the camera is panned far from origin (e.g. world position
   ~1000 voxels away). Today these scenes silently lose lighting
   beyond the static window.
2. Same demos with the camera at origin produce a screenshot diff
   within sampling noise of T-072's reference output — no regression
   on existing scenes.
3. Light-volume texture memory footprint unchanged (still 128³ RGB).
4. `fleet-build --target IRShapeDebug` clean on `linux-debug` AND
   `macos-debug`.
5. Backend parity: GLSL and Metal shaders both read the new origin
   uniform; `render-debug-loop` shows no parity drift.

## Gotchas

- **Snap to grid resolution, not pixels.** Snapping `worldOrigin_`
  to a sub-voxel value re-emits all voxel cells per frame and creates
  per-frame shimmer. Snap to integer voxel multiples (i.e. multiples
  of `kCellSize`).
- **Propagate stays origin-agnostic.** Only seed and consumer need
  `worldOrigin_`.
- **Use the iso camera target, not the window centre.** The trixel
  canvas centre is the right reference; under non-default zoom or
  viewport offset, the GLFW window centre diverges from it.
- **Out-of-volume light culling is by design.** Lights farther than
  the volume's half-extent from the camera contribute nothing — same
  behaviour as today's fixed-origin volume, just relative to the
  camera now. Document as a known limitation: light radius is bounded
  by the volume size regardless of anchoring strategy.
- **T-072 design stamping.** This task assumes T-072 ships a frame-
  data UBO that already exists for the seed/propagate dispatches; the
  origin field rides along on whatever T-072 introduces, not a new
  buffer. If T-072's design ends up storing per-frame light data in
  an SSBO instead, mirror the addition there.
- **Issue scope.** The original `Mechanics` section in #360 names
  `BUILD_OCCUPANCY_GRID` and `c_compute_sun_shadow.glsl`. Both are
  out-of-scope here — they're being deleted, not migrated. PR body
  for this task should call out the rescope explicitly so reviewers
  don't expect occupancy-side changes.
