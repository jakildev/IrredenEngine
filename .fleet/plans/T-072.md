# Issue #359 / T-072 — canonical plan (architect direction)

**Status:** Blocked-by T-071 (PR #423 once it resumes). Plan written
2026-05-02 alongside the design-unblock of T-071.

GPU-side light-volume propagation. Replace the CPU BFS + per-light
allocator churn + 8 MB `subImage3D` upload in
`engine/prefabs/irreden/render/systems/system_compute_light_volume.hpp`
with GPU compute passes.

## v1 architectural choice (the design doc punted on this)

**Direction:** jump-flood / iterative-dilation propagate, with a small
GPU-built voxel-pool-only occupancy structure used purely for LOS
during seeding and propagation. Do **not** attempt projective light-
space depth maps for omni LOS in v1 — that's the long-range unified
substrate but is too big a step inside this task.

Rationale:

1. The existing `system_compute_light_volume.hpp` does flood-fill with
   6-connected occupancy reads. Jump-flood maps to that semantics
   directly: seed at light origin, dilate per pass, mask propagation by
   occupancy. Lift the CPU BFS to GPU compute one-for-one.
2. A small GPU occupancy is cheap to rebuild per frame from
   `C_VoxelPool` data alone (no SDF rasterization — that step was
   deleted in T-071). This is *not* the same buffer as the deleted
   `OccupancyGridBuffer` — sized to the light-volume's 128³ extent, not
   the legacy 256³.
3. SDF occlusion for point/spot LOS (issue #364) is intentionally NOT
   restored in v1. It's a known regression from the lighting-fidelity-
   polish PR and is acceptable until the projective-LOS rewrite lands
   later. v1 plan: keep #364 open, document the gap in the demos that
   need it.

## v1 task shape

1. **GPU occupancy build** — new compute shader
   `c_build_light_volume_occupancy.{glsl,metal}` (or fold into an
   existing compute pass if a natural seam exists). Reads `C_VoxelPool`
   live data (already on GPU via the voxel SSBO), writes a 128³ bitfield
   to a new SSBO sized to the light-volume extent. Runs at the start of
   the lighting stage, before the seed pass.
2. **Seed pass** — new compute shader. For each `C_LightSource +
   C_PositionGlobal3D` entity, write its `emitColor * intensity` into the
   light volume texture at the rounded world position. Spot lights set
   the cone direction + half-angle in a per-light constant buffer for
   the propagate pass to read.
3. **Propagate pass(es)** — new compute shader, ping-pong between two
   light-volume textures. Each invocation reads neighbour cells, applies
   the falloff (1 - d/radius), masks by occupancy, max-composites into
   its own cell. Number of passes = max-light-radius (32 cells today);
   tune for cost. Spot cone test runs per-cell against the per-light
   direction + half-angle.
4. **Sample** — `c_lighting_to_trixel.glsl` continues to sample the
   volume texture. No change.
5. **Delete from CPU** — `detail::floodFillEmissive`,
   `detail::fillPointLight`, `detail::fillSpotLight`,
   `detail::hasLineOfSight`, the `cpuBuffer_` member of
   `C_CanvasLightVolume`, the `subImage3D` upload. The component
   becomes a pure GPU resource holder.

## Acceptance

1. `system_compute_light_volume.hpp` has no per-frame CPU `vector<>`,
   no `std::fill`, no `subImage3D` of the volume texture. CPU work per
   frame ≤ collecting `C_LightSource` entities + uploading a small
   per-light constant buffer.
2. `IRLightingCombined`, `IRLightingEmissive`, `IRLightingPoint`,
   `IRLightingSpot` produce output within sample-noise of pre-migration
   reference (capture before T-072 lands; compare via
   `render-debug-loop`).
3. `fleet-build --target IRShapeDebug` clean on `linux-debug` AND
   `macos-debug`.
4. Issue #364 (SDF LOS) explicitly noted in the PR body as still open;
   no regression introduced beyond what already exists.

## Out of scope (deferred)

- **Projective light-space depth maps for omni LOS** — the long-range
  unified substrate. Will replace the GPU occupancy bitfield in a
  later task once the screen-space-everything pattern is mature on
  more lighting paths. NOT v1.
- **SDF occlusion in point/spot LOS** (issue #364) — restore in
  whichever later task adds projective LOS, OR as a smaller
  intermediate that adds SDF evaluation to the propagate pass's cell
  occupancy test.
- **Camera-anchored grid** (issue #360) — independent. Can land
  before, after, or alongside T-072. Author's choice.
- **Dirty-tracked re-runs** — moot once propagation is GPU-side; per-
  frame cost is already a small fixed handful of dispatches.

## Open questions for the implementing agent

1. Workgroup size for the propagate pass — 8×8×4? 16×16×1? Benchmark
   on Apple M-series and Intel/AMD before committing.
2. Number of propagate passes — fixed at max-light-radius (32) or
   dynamic per max active light? Probably fixed for v1 (simpler).
3. Should the GPU occupancy build be a fresh dispatch per frame or
   ping-ponged with a dirty flag? Default fresh; only optimize if a
   profile shows it matters.
