Source: https://github.com/jakildev/IrredenEngine/issues/289 (comment by jakildev, 2026-04-25)

## Overview

Complete the stacked-PR system started in T-020. Five scope items, each its own
PR, executed in stack order so the work dogfoods the system it is implementing.

## T-041: commit-and-push skill — complete stack-aware mode

**Files:** `.claude/skills/commit-and-push/SKILL.md`

- Detect active molecule via `fleet-claim molecule show <agent>` (or rely on
  `stack-base` returning a non-master branch).
- Add `fleet:stacked` label when `--base != master`.
- Standardize PR body with `Stack context:` listing the previous PR (or
  `master`) and the full chain.

**Acceptance:** opening a PR with an active molecule produces `--base
<prev-branch>`, `fleet:stacked` label, body with stack context. No-molecule
case still produces `--base master`.

**Size:** small (~40 LOC docs).

## T-042: start-next-task skill — stack-aware reset

**Depends on:** T-041

**Files:** `.claude/skills/start-next-task/SKILL.md`

Currently the skill says "Do NOT branch off your previous PR branch". Fix:
- Call `fleet-claim molecule resume <agent>` before resetting.
- If stdout names a `T-NNN`, branch off the just-opened PR's head ref,
  not `origin/master`.
- If stdout is empty, keep current behavior (reset to master).

**Acceptance:** after opening PR for first task in a 2-task stack, running
`start-next-task` lands on a branch whose merge-base is the just-opened PR's
head ref. After exhausting the stack, resets to `origin/master` as today.

**Size:** small (~30 LOC docs).

## T-043: reviewer upstream approval gating

**Depends on:** T-042

**Files:**
- `scripts/fleet/fleet-labels` — add `fleet:awaiting-upstream-review`
- `.claude/commands/role-sonnet-reviewer.md`
- `.claude/commands/role-opus-reviewer.md`

Reviewer detects `baseRefName != "master"`, looks up upstream PR by head ref:
- Upstream OPEN with `fleet:needs-fix`/`fleet:blocker`: add
  `fleet:awaiting-upstream-review`, comment, skip verdict.
- Upstream OPEN with `fleet:approved`: proceed with own-diff review.
- Upstream MERGED: per #287, merger re-targeted; proceed normally.
- Upstream not found / closed-not-merged: comment, skip, surface to human.

**Acceptance:** open downstream while upstream has `fleet:needs-fix`
→ `fleet:awaiting-upstream-review` applied, no `fleet:approved`. After
upstream re-approved, next reviewer pass proceeds normally.

**Size:** small (~50 LOC docs + 1 label entry).

## T-044: downstream auto-rebase when upstream changes

**Depends on:** T-043

**Files:**
- `scripts/fleet/fleet-claim` — add `molecule rebase-downstream` subcommand
- `.claude/commands/role-sonnet-author.md`
- `.claude/commands/role-opus-worker.md`

New subcommand reads molecule state, rebases each downstream branch onto the
new upstream tip in stack order, force-pushes with `--force-with-lease`.
Author role's "address feedback" sub-flow invokes this after pushing the
upstream fix and comments on each downstream PR.

Conflict handling: abort rebase, comment on affected PR, add `fleet:blocker`,
surface to human. Do NOT attempt heuristic resolution.

**Acceptance:** stack A→B→C; reviewer flags A with `fleet:needs-fix`; worker
fixes A, pushes, runs rebase subcommand; B and C branches now have new A tip
as parent. Conflicts surface as `fleet:blocker` + comment.

**Size:** medium (~80 LOC; subcommand is the bulk).

## Gotchas

- Don't bundle. One scope item per PR.
- Source of truth = molecule state, not labels or commit subjects.
- `fleet-claim molecule resume` empty stdout distinguishes "nothing to
  resume" from "more work" via stderr — T-042's logic must read both.
- `--force-with-lease`, never `--force` in T-044's rebase subcommand.
- Conflict in downstream rebase = stop, don't auto-resolve.
- T-043's gate must not flip `fleet:approved` to awaiting on a second-pass
  review. Check current label state before applying.
- T-041 opens against `master` (bootstrapping); T-042 onward chains.

## Full acceptance test (parent issue level)

A 5-PR stack opened by an opus-worker:
- Workers open all 5 chained PRs cleanly (T-041 + T-042).
- Reviewer holds approval on PRs 2-5 until PR 1 is approved (T-043).
- Reviewer flags PR 1 with `fleet:needs-fix`; worker fixes; PRs 2-5
  auto-rebase onto new PR 1 tip (T-044).
- Reviewer re-approves PR 1; PRs 2-5 become reviewable in order.
- All 5 approve; merger (per #287) cascades onto master.
- No manual rebase, no manual label fix, no manual base re-target.
