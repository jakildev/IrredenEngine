# Plan: #302 — Generic modifier framework

## Context

- Parent issue: https://github.com/jakildev/IrredenEngine/issues/302
- Scope: an engine-level modifier framework letting any component
  field be modulated by a vector of modifiers, with a dense ECS-native
  resolver pipeline.

The pattern is conceptually similar to `C_Position3D` +
`C_PositionOffset3D` → `C_PositionGlobal3D`: a "base" value lives on
one component, modifications live on another, a resolver computes the
effective value, and reader systems read the resolved cache. This
plan generalizes that pattern.

## Locked design choices (decided during planning)

After two design passes, the locked-in shape:

### Transform shape — Path C, hybrid

Structured `TransformKind` covers the common case
(`ADD` / `MULTIPLY` / `SET` / `CLAMP_MIN` / `CLAMP_MAX` / `OVERRIDE`).
Lambda escape hatch in a sibling component for the long tail.

### Composition pipeline — eager, no dirty flag

Resolver runs every tick over a dense archetype iteration. Recompute
is O(modifiers per entity) per frame; for typical entities (~5
modifiers) this is <1 cache line of work. No dirty flags — the
branching cost outweighs the savings, and the ECS is designed for
this.

### Storage shape — vector-on-component

`C_Modifiers` carries `std::vector<Modifier>`. Dense, cache-friendly,
trivially-copyable element type. Entity-per-modifier is a fine v2
optimization for cases that genuinely need modifier-as-entity (own
particle effect, own physics, etc.) but doesn't earn its complexity
in v1.

### Source attribution — `EntityId`, not strings

Each modifier carries `EntityId source_`. Engine-native, dense, gives
human-readable names "for free" via the source entity's `C_Name`.
Source-tied lifetime falls out: when the source dies, sweep
modifiers where `source_ == destroyedId`.

### Decay — `ticksRemaining_` only

Built-in lifetime is just a tick counter and a one-time sweep of
expired modifiers. Non-linear / curved / source-driven decay is
**not the modifier's problem** — the source entity's tick function
adjusts its modifier's `param_` directly, or destroys-and-replaces.

### Globals + exemption — singleton component + archetype tag

A singleton entity carries `C_GlobalModifiers`. Exemption is a
`C_NoGlobalModifiers` tag. Resolver dispatch is via **archetype
routing** (two registered systems, one per archetype with-tag /
without-tag), not per-entity branching.

## Reference data shapes

```cpp
using FieldBindingId = std::uint16_t;  // dense registry index

enum class TransformKind : std::uint8_t {
    ADD, MULTIPLY, SET, CLAMP_MIN, CLAMP_MAX, OVERRIDE,
};

struct Modifier {
    FieldBindingId field_;          // what is being modified
    TransformKind  kind_;
    float          param_;          // amount, multiplier, set value, clamp bound
    EntityId       source_;         // who pushed this
    int32_t        ticksRemaining_; // -1 = no decay; 0 = expired
};
// Trivially-copyable, ~16 B + alignment.

struct C_Modifiers       { std::vector<Modifier> modifiers_; };
struct C_GlobalModifiers { std::vector<Modifier> modifiers_; };  // singleton
struct C_NoGlobalModifiers {};                                    // empty tag

struct LambdaModifier {
    FieldBindingId              field_;
    std::function<float(float)> fn_;
    EntityId                    source_;
    int32_t                     ticksRemaining_;
};
struct C_LambdaModifiers { std::vector<LambdaModifier> modifiers_; };
```

## Resolver pipeline

Once per pipeline execution, end of UPDATE phase, before RENDER reads:

```
ModifierDecay              decrement ticksRemaining_, drop expired
GlobalModifierDecay        same on the singleton
ModifierResolveGlobal      non-exempt entities (archetype filter)
ModifierResolveExempt      exempt entities (with C_NoGlobalModifiers)
ModifierResolveLambda      lambda escape hatch, both archetypes
→ RENDER reads C_ResolvedFields directly
```

5 systems, all dense archetype iterations, no hash-map lookups in any
tick body.

## Existing patterns to migrate (audit)

The engine already has hand-rolled `base + modifier → resolved`
patterns. The framework's first internal consumers:

**v1 migration targets (in this epic):**

- **Position pattern** — `C_Position3D` + `C_PositionOffset3D` →
  `C_PositionGlobal3D`. Canonical example, the worked case in the
  issue body. `system_apply_position_offset.hpp`,
  `system_update_positions_global.hpp` get reframed (or replaced) by
  the new resolver.
- **Velocity drag** — `C_VelocityDrag` modulates `C_Velocity3D`. A
  clean MULTIPLY transform on a velocity field.
  `system_velocity_drag.hpp` gets reframed.

**Follow-up candidates (post-epic):**

- **Animation color** (`C_AnimColorState` + `system_animation_color.hpp`)
  — modulates color over time via a curve. Might want lambda escape
  hatch.
- **Spring color** (`system_spring_color.hpp`) — physics-driven color
  modulation. Lambda territory.
- **Spawn glow / trigger glow** — time-bounded visual modifiers on
  brightness / glow.
- **Texture scroll** — UV offset modulation. Less obviously a
  modifier (it's input to UV computation, not a target value), so
  may not migrate.

Listed in child 1's design doc as "future migration candidates." Each
gets its own follow-up task once the framework has baked.

### Child 2 — Core framework runtime `[opus]`

Area: `engine/prefabs/irreden/common/`, `engine/system/`.

Output:
- `FieldBindingId` registry: register-at-init API, dense lookup.
- `C_ResolvedFields` reset / apply machinery.
- 5 resolver systems wired into the pipeline:
  - `ModifierDecay`
  - `GlobalModifierDecay`
  - `ModifierResolveGlobal` (non-exempt archetype)
  - `ModifierResolveExempt` (exempt archetype)
  - `ModifierResolveLambda`
- Pipeline registration helper that registers all five in correct
  order.
- Source-destruction sweep helper (cleanly removes modifiers whose
  `source_` equals a destroyed entity).
- `applyToField(entityId, fieldId, baseValue) -> float` direct query
  (used by readers that don't want to maintain their own resolved
  cache).

Acceptance:
1. Unit tests cover: ADD / MULTIPLY / SET / CLAMP composition
   correctness, ticksRemaining_ decay, source-destruction sweep,
   global + exemption dispatch.
2. No `getComponent` / `getComponentOptional` calls inside any tick
   body (verified by reading the system source).
3. Profile of resolver tick at 1000 entities × 5 modifiers each
   < 0.5 ms (CPU profiler). Recorded in PR body.
4. Builds clean on `linux-debug` AND `macos-debug`.

Depends on: child 1.

   physics demo if one exists; otherwise add a 30-line test

   the framework absorbs the one-off pattern.

### Child 5 — `modifier_demo` creation: visual showcase `[sonnet]`

Area: `creations/demos/modifier_demo/`, docs.

A runnable visual demo that exercises every framework capability with
on-screen labels showing what's happening. Each capability is
key-toggled so the human can see the effect at the press of a key.

**Demo layout.** A row of voxel cubes (or sprites once #14 ships)
moving along a horizontal axis. Each cube has `C_Position3D` +
`C_Velocity3D` + `C_Modifiers`. The framework's effects are visible
because the cubes move (or stop, or speed up, or oscillate) under
modifier influence.

**On-screen HUD** (text overlay):
- Each cube's resolved speed (so the math is visible).
- Active modifiers per cube with source and ticksRemaining.
- A legend mapping number keys to capabilities.

**Capabilities exercised:**

- **Press 1 — Haste (MULTIPLY 1.5×, 300 ticks):** cubes visibly
  speed up, then return to normal. Showcases structured MULTIPLY
  + ticksRemaining decay.
- **Press 2 — Stun (SET 0, 180 ticks):** cubes freeze. Showcases
  SET transform.
- **Press 3 — Slow (MULTIPLY 0.3×, 300 ticks):** cubes crawl.
- **Press 4 — Stack (Haste + Slow at once):** cubes move at
  1.5×0.3 = 0.45× speed. Showcases composition.
- **Press 5 — Global Slow (singleton, MULTIPLY 0.5×, 600 ticks):**
  ALL cubes slow, except one tagged with `C_NoGlobalModifiers`
  (clearly labeled "exempt"). Showcases global + archetype-routed
  exemption.
- **Press 6 — Lambda Sinusoidal:** cube's speed oscillates via
  `1.0 + 0.5 * sin(t)` lambda. Showcases the lambda escape hatch.
- **Press 7 — Source Kill:** destroys the modifier source entity
  for one cube; that cube's modifiers vanish (others persist).
  Showcases source-tied lifetime.
- **Press 8 — Clamp:** apply `CLAMP_MAX 0.5` so a Haste-buffed
  cube can't exceed 0.5×. Showcases clamp transforms.

**All wiring in Lua**, exercising the bindings from child 4 — the
demo doubles as the framework's integration test.

Documentation:
- Brief `creations/demos/modifier_demo/README.md` explaining the
  demo and how to run it.
- Add to `creations/CLAUDE.md` roster.
- `--auto-screenshot` shot list capturing one screenshot per
  capability (or a curated subset that's visually distinct).

Acceptance:
1. `fleet-run IRModifierDemo` launches and shows a row of moving
   cubes.
2. Each numbered key triggers the corresponding capability; the
   visual change is obvious.
3. The on-screen HUD shows resolved values that match the active
   modifiers.
4. `fleet-run IRModifierDemo --auto-screenshot 12` produces the
   committed shot list.
5. Builds clean on `linux-debug` AND `macos-debug`.

Depends on: children 2, 3, 4.
