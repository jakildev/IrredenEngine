# Plan: ECS singleton-component infrastructure (#646)

> **Companion plan:** the full architecture lives in `~/.fleet/plans/issue-200.md` (engine time substrate). This file is **Task A** of that plan, filed separately as the prerequisite. The sim-clock work in #200 consumes this API.

## Context

The engine has one production singleton-shaped use case today: `C_GlobalModifiers` on a `"modifierGlobals"` entity, with a hand-rolled cached-ID lookup via `detail::globalsEntityId()` and `currentGlobalModifiersPtr()` (`engine/prefabs/irreden/common/modifier.hpp:185-191`). Every future "global game state component" wants the same shape: lazy-init on first access, one canonical entity per type, cached ID lookup.

Today every singleton-shaped case has to invent that lazy-init pattern itself. The modifier framework's version is fine but invisible from outside (a `detail::` helper, no public API). The next person needing a singleton has nothing to copy.

This task generalizes the pattern into an explicit `IRECS::` API and retrofits the modifier framework to use it as validation. Once it lands, #200 (sim-clock substrate) can layer on top cleanly.

## API surface

```cpp
namespace IRECS {
    template <typename C>
    C& singleton();              // lazy-init; returns ref to THE component

    template <typename C>
    C* singletonOrNull();        // no-create variant

    template <typename C>
    IREntity::EntityId singletonEntity(); // for systems iterating by archetype
}
```

**Semantics:**
- First call to `singleton<T>()` creates a named entity `"singleton:<typeid>"` with default-constructed `T`. Cached entity ID per type via `std::unordered_map<std::type_index, EntityId>` in the entity manager.
- Subsequent calls return the same reference (cache hit, no archetype walk).
- Systems can iterate `C_X` as usual — singleton entities are real ECS entities that match the archetype.
- Lua: `ir.entity.singleton(componentTypeId)` returns the singleton as a Lua table.

## Scope

1. The three API entry points above.
2. Lazy-init mechanics in the entity manager.
3. Lua binding mirroring existing component reflection conventions.
4. **Retrofit `C_GlobalModifiers`** to use the new API. Replaces `detail::globalsEntityId()` + `currentGlobalModifiersPtr()`.
5. **Documentation** in `engine/entity/CLAUDE.md` — when to use, naming convention, archetype implications, save/load implications.

## Acceptance

1. `IRECS::singleton<T>()` returns a stable reference across calls.
2. Modifier framework's globals access uses the new API; behavior identical; modifier-framework tests pass unchanged.
3. Lua test script creates a singleton component, reads/writes it.
4. `engine/entity/CLAUDE.md` documents the API + conventions.
5. Builds clean on `linux-debug` and `macos-debug`.

## Out of scope

- Singleton-component save/load integration — depends on #199 (not merged). Retrofit via #199 follow-up.
- Compile-time enforcement of singleton-only-ness via a trait — runtime convention sufficient for v1.
- Refactoring `RenderManager`'s designated entities (`m_mainCanvas`, `m_camera`, etc.) — they're already cached on the manager; refactoring is a separate cleanup.

## Verification (end-to-end)

1. **Build clean.** `fleet-build` on `linux-debug` and `macos-debug`.
2. **API stability.** Unit test creates a `C_TestSingleton`, calls `IRECS::singleton<C_TestSingleton>()` twice, asserts both return the same reference. Mutates the singleton, calls again, asserts mutation persists.
3. **Retrofit validation.** Existing modifier-framework tests pass with the new API in place (no behavior change observable from outside).
4. **Lua round-trip.** Lua script creates a singleton with `ir.entity.singleton(...)`, reads a field, writes a field, reads again, asserts mutation persisted.

## Notes for the worker

- The lazy-init pattern in `modifier.hpp:185-191` is the reference shape. The new API is its public form.
- Naming: `IRECS::singleton<T>()` is the working name. Worker can adjust to fit existing namespace conventions (`IREntity::singleton<T>()` or `IREntity::global<T>()` are reasonable alternatives) — flag the choice in the PR body.
- Don't over-engineer. The lazy-init pattern is ~30 lines of real code; the public surface is ~10 lines. Most of the work is the Lua binding and the doc.
